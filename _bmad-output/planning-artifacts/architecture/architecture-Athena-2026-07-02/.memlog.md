- (event) Architecture create run started from final Athena PRD; coaching path selected; spine-only deliverable requested.
- (decision) M0 adopted as first proof slice: minimal Electrical/Runtime DSL (20-30 core keywords) -> AST -> semantic validation (references, types, ports, connections) -> stable Engineering IR -> simple SVG renderer -> 5-10 examples. AI, OPC UA, and product-specific functionality are explicitly out of scope for this spine's initial proof.
- (decision) AD candidate accepted in coaching: M0 uses a single-process compiler substrate with strict internal module boundaries; no service-shaped or distributed architecture at proof stage.
- (decision) AD candidate accepted in coaching: M0 keeps a small general semantic core and implements electrical/runtime as the first domain extension; the first wedge must not become the permanent core vocabulary.
- (decision) AD candidate accepted in coaching: Engineering IR owns semantic truth; a separate render-facing model owns view truth; SVG is a backend only and may not recover or invent semantics.
- (decision) AD candidate accepted in coaching: AST remains syntax-level only; semantic validation and normalization produce the canonical Engineering IR; rule execution authority for M0 belongs only to Engineering IR, never to AST.
- (decision) AD candidate accepted in coaching: M0 has one and only one canonical semantic model, Engineering IR; normalization is a compiler pass, not a second durable IR layer.
- (decision) AD candidate accepted in coaching: examples/ are conformance artifacts for M0; each example defines stable contract inputs plus expected validation, IR shape, and SVG output class, and changes are architectural rather than casual fixture churn.
- (decision) AD candidate accepted in coaching: M0 must prove a real plugin system rather than only internal extension interfaces; plugin architecture is part of the first proof, not deferred.
- (decision) AD candidate accepted in coaching: M0 uses typed plugin classes rather than a generic plugin hook; at minimum domain, renderer, and rule/plugin-extension contracts are distinct and all depend inward on core contracts only.
- (constraint) AutomationML is a load-bearing standards boundary: it may enter through explicit compiler and typed plugin contracts as an import/export boundary for industrial semantic exchange, but it may not define the native internal model or ontology wholesale.
- (decision) AutomationML use refined in coaching: for M0 it is a standards and ontology reference concept, not a required implemented plugin target and not a replacement for Athena's internal model; it informs future boundary design only.
- (decision) AD candidate accepted in coaching: plugins may extend the system only through core-owned contracts; domain, rule, and renderer plugins may not privately redefine or mutate the meaning of Engineering Language, Engineering IR, or compiler semantics.
- (decision) AD candidate accepted in coaching: M0 seed stack stays Kotlin-centered and single-language for proof stage; Kotlin/JVM owns DSL, AST, validation, Engineering IR, compiler passes, plugin contracts, CLI execution, and initial SVG rendering; no Rust split, Compose, or WASM in M0.
- (decision) Stack decision refined in coaching: M0 is JVM-first, not just Kotlin-centered. The proof substrate optimizes for compiler clarity and CLI execution on Kotlin/JVM before any multiplatform concerns enter the architecture.
- (decision) AD candidate accepted in coaching: M0 uses explicit JVM module boundaries: language, semantics-core, domain-electrical-runtime, ir, compiler, renderer-svg, examples, and cli. Dependency direction stays inward; cli orchestrates, compiler owns passes, plugins depend on core contracts only.
- (decision) AD candidate accepted in coaching: M0 real plugins use typed manifest metadata plus local classpath discovery with compatibility validation before use; remote distribution and dynamic marketplace concerns remain out of scope.
- (decision) AD candidate accepted in coaching: semantic/layout/geometry separation is binding from M0 onward. Engineering IR remains semantic only; any positioning logic for SVG lives in a thin render-facing layer; a full durable Layout IR is deferred until view logic becomes substantial.
- (version) M0 JVM baseline pinned to Java 25 LTS; seed-stack rule clarified as latest stable LTS at authoring time, then explicitly pinned rather than floating 'always latest'. Kotlin 2.4.0 and Gradle 9.6.1 align with this seed as of 2026-07-02.
