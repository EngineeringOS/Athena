# Athena - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for Athena M8, decomposing the unified semantic mutation model requirements from the PRD and architecture into implementable stories.

## Requirements Inventory

### Functional Requirements

FR-1: Athena can keep one mutation authority for both text and graph interaction.
FR-2: Athena can classify meaningful changes explicitly.
FR-3: Athena can translate supported graph operations into Athena-owned command intents.
FR-4: Athena can refresh graphical state deterministically after an accepted change.
FR-5: Athena can route accepted mutation outcomes into one review model.
FR-6: Athena can support coherent reveal and navigation across representations.
FR-7: Athena can define what each projection may display, edit, emit, and own.
FR-8: Athena can keep mutation semantics independent from the current renderer stack.

### NonFunctional Requirements

NFR-1: Meaningful changes must route through one Athena-owned mutation path.
NFR-2: Canonical engineering meaning remains upstream of any renderer or editor client.
NFR-3: The same accepted mutation over the same state yields the same resulting canonical and projection state.
NFR-4: Command intents, mutation outcomes, rejection paths, and review facts must remain inspectable for development and architecture debugging.
NFR-5: Graph-originated and source-originated mutations must share one semantic review and history vocabulary.
NFR-6: The current graph stack may implement interaction delivery, but it must not own command meaning or durable mutation semantics.

### Additional Requirements

- M8 builds on completed M1 command/runtime ownership, completed M6 semantic review/history authority, and completed M7 projection and graph-workbench boundaries and must not reopen those milestone centers.
- Accepted source-originated and graph-originated changes must converge into the same Athena-owned command, validation, and refresh path.
- M8 must classify interaction as semantic mutation, projection mutation, or transient interaction instead of inferring category from UI behavior.
- Graph gestures must emit Athena command intent rather than renderer-native save logic.
- Runtime must own acceptance, rejection, validation feedback, and deterministic refresh after mutation.
- Accepted mutation outcomes must feed the same semantic diff/review/history vocabulary already owned by `kernel/semantic-scm`.
- Bidirectional reveal across source, graph, and semantic SCM must anchor through canonical semantic identity and governed projection references.
- Every interactive projection must publish an explicit ownership contract for display, editability, command emission, transient state, and persisted projection metadata.
- M8 remains narrow: one real semantic mutation path, one real projection mutation path, one rejection/validation feedback path, and one reveal/review path shared across source and graph.

### UX Design Requirements

None identified. M8 is a mutation-authority and review-coherence milestone, not the final visual design-system or notation-depth milestone.

### FR Coverage Map

FR-1: Epic 1 - unified mutation authority and runtime mutation contract
FR-2: Epic 1 - mutation categories and projection ownership contracts
FR-3: Epic 2 - graph gesture to Athena command-intent routing
FR-4: Epic 2 - accepted mutation refresh and rejection snapback behavior
FR-5: Epic 3 - unified semantic review facts
FR-6: Epic 3 - source/graph/review reveal and anchoring
FR-7: Epic 1 - explicit projection ownership contracts
FR-8: Epic 2 - renderer-neutral mutation semantics preserved through graph integration

## Epic List

### Epic 1: Freeze The Unified Mutation Authority
Platform engineers can expose one Athena-owned mutation authority for both source and graph while classifying semantic mutation, projection mutation, and transient interaction explicitly.
**FRs covered:** FR-1, FR-2, FR-7

### Epic 2: Route Graph-Originated Editing Through Athena Commands
Engineers can perform one supported semantic edit and one supported projection edit from the graph workbench and have runtime accept, reject, validate, and refresh them without renderer-owned truth.
**FRs covered:** FR-3, FR-4, FR-8

### Epic 3: Unify Review And Reveal Across Source And Graph
Reviewers and engineers can understand accepted changes through one semantic review model and move coherently between source, graph, and semantic SCM context after either source-originated or graph-originated mutation.
**FRs covered:** FR-5, FR-6

## Epic 1: Freeze The Unified Mutation Authority

Platform engineers can expose one Athena-owned mutation authority for both source and graph while classifying semantic mutation, projection mutation, and transient interaction explicitly.

### Story 1.1: Publish Unified Mutation Contracts And Categories In Runtime-Owned Command Surfaces

As a platform engineer,
I want Athena to define explicit mutation contracts and categories,
So that source and graph change requests converge into one inspectable runtime-owned mutation model.

**Acceptance Criteria:**

**Given** the completed M1 command runtime and the M8 architecture spine
**When** M8 begins implementation
**Then** Athena publishes explicit mutation contracts for accepted, rejected, and validation-feedback outcomes
**And** those contracts remain Athena-owned rather than renderer- or editor-owned

**Given** interaction types are reviewed
**When** mutation categories are checked
**Then** Athena classifies supported behavior as semantic mutation, projection mutation, or transient interaction
**And** the category is carried explicitly through the mutation model instead of being inferred from frontend behavior

### Story 1.2: Define Projection Ownership Contracts For The First Interactive Projections

As a platform engineer,
I want each interactive projection to declare what it may display, edit, emit, and own,
So that editability is governed explicitly instead of emerging accidentally from UI capability.

**Acceptance Criteria:**

**Given** the current supported projections such as `cabinet` and `wiring`
**When** Athena marks them as interactive or non-interactive in M8
**Then** each projection publishes an ownership contract for display, editability, command emission, transient state, and persisted projection metadata
**And** unsupported mutation attempts are rejected or snapped back on refresh

**Given** projection ownership is reviewed against kernel boundaries
**When** dependency direction is checked
**Then** the contracts stay renderer-neutral and domain-extensible
**And** they do not move engineering truth into layout or renderer-local state

### Story 1.3: Normalize Source-Originated Changes Into The Same Runtime Mutation Result Path

As an IDE and runtime engineer,
I want source-originated edits to produce the same governed mutation-result shape used by graph-originated edits,
So that text and graph remain two clients of one mutation authority before review and persistence.

**Acceptance Criteria:**

**Given** an accepted source-originated change in an active Athena repository session
**When** Athena evaluates the change
**Then** runtime produces the same governed mutation-result structure used by graph-originated change requests
**And** validation, refresh, and later review consequences are represented through the same runtime-owned path

**Given** a source-originated change fails validation or is rejected by ownership policy
**When** the result is surfaced
**Then** Athena exposes inspectable rejection or validation-feedback output
**And** the failure does not create a second editor-only mutation semantics path

## Epic 2: Route Graph-Originated Editing Through Athena Commands

Engineers can perform one supported semantic edit and one supported projection edit from the graph workbench and have runtime accept, reject, validate, and refresh them without renderer-owned truth.

### Story 2.1: Translate Supported Graph Gestures Into Athena Command Intents

As a platform engineer,
I want the graph workbench to emit Athena command intents instead of renderer-native save behavior,
So that graph interaction remains downstream of Athena mutation semantics.

**Acceptance Criteria:**

**Given** the current Theia graph workbench and `integrations/graph-glsp` adapter
**When** users perform a supported graph-originated action
**Then** the frontend and adapter translate that action into Athena-owned command intent over existing transport seams
**And** no renderer-local save model becomes persistent authority

**Given** command-intent payloads are reviewed
**When** inspectability is checked
**Then** intent identity, mutation category, and target subject remain visible enough for debugging and architecture review
**And** the graph stack remains a downstream client of Athena command meaning

### Story 2.2: Prove The First Semantic Mutation Path From The Graph Workbench

As an engineer,
I want one supported graph-originated semantic edit to execute through Athena runtime,
So that the graph becomes a real engineering editing surface without bypassing validation or semantic authority.

**Acceptance Criteria:**

**Given** a supported graph-originated semantic edit path
**When** the user requests that edit from the graph workbench
**Then** Athena routes the request through command intent, runtime evaluation, validation, canonical engineering-state update, and deterministic projection refresh
**And** the resulting graph state reflects the accepted semantic outcome rather than local canvas truth

**Given** the same semantic edit is invalid or disallowed
**When** runtime rejects it or returns validation feedback
**Then** the workbench surfaces that result through Athena-owned feedback
**And** the graph does not retain durable divergent local state

### Story 2.3: Prove The First Projection Mutation Path From The Graph Workbench

As an engineer,
I want one supported graph-originated projection edit to execute through Athena runtime,
So that layout or projection changes remain governed without being confused with engineering truth.

**Acceptance Criteria:**

**Given** a supported graph-originated projection mutation such as node position or grouping adjustment
**When** the user performs that action in the graph workbench
**Then** Athena routes the request through governed command semantics into projection/layout metadata update
**And** projection refresh reflects the accepted metadata change deterministically

**Given** a projection mutation is unsupported, invalid, or outside the projection ownership contract
**When** Athena refreshes the affected view
**Then** the unapproved local state snaps back or is discarded
**And** the rejection path remains inspectable and renderer-neutral

## Epic 3: Unify Review And Reveal Across Source And Graph

Reviewers and engineers can understand accepted changes through one semantic review model and move coherently between source, graph, and semantic SCM context after either source-originated or graph-originated mutation.

### Story 3.1: Feed Accepted Mutation Outcomes Into One Semantic Review Model

As a reviewer,
I want accepted mutations from either source or graph to produce the same semantic review facts,
So that review meaning stays semantic rather than splitting by interaction origin.

**Acceptance Criteria:**

**Given** an accepted mutation originated from source or graph
**When** Athena computes change consequences
**Then** semantic diff, review summary, and history vocabulary are produced through the existing M6 semantic review path
**And** graph-originated changes do not invent a renderer-specific review language

**Given** review outputs are inspected for mixed interaction origins
**When** review semantics are compared
**Then** the same engineering change yields compatible review facts regardless of whether the initiating surface was source or graph
**And** `kernel/semantic-scm` remains the downstream review/history authority

### Story 3.2: Support Bidirectional Reveal Across Source, Graph, And Semantic SCM Context

As an engineer or reviewer,
I want source, graph, and semantic SCM views to reveal the same accepted change coherently,
So that I can move between representations without identity drift.

**Acceptance Criteria:**

**Given** source, graph, and semantic SCM surfaces are open together
**When** the user reveals a changed subject from any one of those surfaces
**Then** Athena resolves the corresponding graph, source, and review context through canonical semantic identity and governed projection references
**And** the same repository and semantic state yield the same reveal outcome

**Given** graph-local ids and projection-local refs exist
**When** reveal anchoring is reviewed
**Then** those ids remain downstream aliases only
**And** canonical semantic identity remains the authority for cross-surface anchoring

### Story 3.3: Publish The M8 Proof Corpus And Verification Path

As an architecture owner,
I want Athena to publish repeatable proof artifacts for the unified mutation model,
So that M8 closes with runnable evidence rather than only design intent.

**Acceptance Criteria:**

**Given** the first semantic mutation path, projection mutation path, and reveal/review path are implemented
**When** Athena publishes the M8 proof corpus
**Then** the milestone includes examples, runnable verification, or equivalent proof artifacts that demonstrate one mutation authority across source and graph
**And** the proof remains narrow instead of widening into broad graphical authoring

**Given** M8 is reviewed as a milestone proof
**When** architecture and product owners inspect the corpus
**Then** it is clear which flows were proven, which feedback/rejection paths exist, and which broader editing concerns remain deferred
**And** later milestones can build on the unified mutation model without reopening its core invariants
