# Athena - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for Athena M7, decomposing the graphical projection and visual workbench requirements from the PRD and architecture into implementable stories.

## Requirements Inventory

### Functional Requirements

FR-1: Athena can define a stable projection boundary for graphical workbench delivery.
FR-2: Athena can keep graphical projection downstream of semantic authority.
FR-3: Athena can host graphical views in the existing Athena workbench.
FR-4: Athena can support graphical navigation and projection-oriented inspection.
FR-5: Athena can define explicit read-only versus editable projection rules.
FR-6: Athena can preserve deterministic graphical refresh from the same underlying semantic state.
FR-7: Athena can prepare for later graphical interaction without turning M7 into a full editing milestone.
FR-8: Athena can choose or validate the graphical technology path at M7 time.

### NonFunctional Requirements

NFR-1: Graphical surfaces must remain downstream of canonical semantic, repository, package, and SCM meaning.
NFR-2: The same upstream semantic state and chosen view must yield the same projection state.
NFR-3: Projection boundary output must remain inspectable for development and architecture debugging.
NFR-4: M7 must extend the current Athena shell rather than replacing it.
NFR-5: M7 must prepare later richer graphical interaction without collapsing into unrestricted editing.
NFR-6: M7 must choose or validate its graphical architecture from current constraints, not from draft-era momentum alone.
NFR-7: View definitions, symbol packs, and renderer assets must remain downstream of the engineering object graph and must not become semantic authority by convention.

### Additional Requirements

- M7 builds on completed M2 projection layers and the completed M4 to M6 IDE and semantic SCM seams and must not reopen those milestone centers.
- The kernel must introduce a dedicated `projection-model` boundary above `layout-model` and `geometry-model` rather than letting renderer clients consume raw engineering entities directly.
- Canonical engineering truth remains the engineering object graph; view definitions and renderer assets remain downstream projection concerns.
- Layout, routing, grouping, and viewport state remain projection metadata and must be invalidated and rebuilt deterministically when upstream semantic or repository state changes.
- The graphical workbench consumes runtime-owned projection sessions through Athena-owned transport rooted at `ide/lsp`.
- That transport is limited to typed projection queries, deterministic projection updates, and an explicit allowlist of governed commands.
- Any graph-framework or protocol adapter remains translation-only in `integrations/graph-*` and must not synthesize engineering truth or persist local state as authority.
- M7 interaction is inspect-first: selection, focus, reveal, and navigation are allowed; persisted projection changes must route through governed runtime commands.
- The first renderer proof must remain relationship-forward and useful without notation-specific asset packs.
- The first renderer proof must also present a graph-first split workbench posture where the active graphical surface is primary, the authored `.athena` source remains visible as secondary context, and broad text-heavy dashboard layout is avoided.
- The first renderer surface should behave like a professional infinite-canvas diagram editor with pan/zoom-oriented navigation rather than a bounded preview panel.
- Domain projection contributions must enter through `extensions/domain-*` and hosted plugin contracts rather than kernel forks.

### UX Design Requirements

None identified. M7 is a platform and architecture proof for graphical projection, not the final visual design-system milestone.

### FR Coverage Map

FR-1: Epic 1 - stable projection-model contracts, runtime sessions, and protocol boundary
FR-2: Epic 1 - semantic authority preserved and view assets kept downstream
FR-3: Epic 2 - graphical workbench delivery inside the current Athena shell
FR-4: Epic 2 - synchronized graphical navigation and projection-oriented inspection
FR-5: Epic 2 - inspect-first interaction rules and governed mutation path
FR-6: Epic 1 - deterministic refresh and invalidation of projection state
FR-7: Epic 3 - first renderer proof and carry-forward interaction discipline
FR-8: Epic 3 - technology-path validation and proof publication

## Epic 1: Freeze The Projection Boundary And Runtime Authority

Platform engineers can expose graphical projection through one deterministic `projection-model` and runtime session path without letting renderer clients, adapters, or frontend state become semantic authority.

### Story 1.1: Publish Canonical Projection Contracts In `kernel/projection-model`

As a platform engineer,
I want Athena to define typed projection contracts in a dedicated kernel boundary,
So that graphical delivery consumes one renderer-neutral projection model instead of raw engineering entities or renderer-specific shapes.

**Acceptance Criteria:**

**Given** the completed M2 layout and geometry layers and the M7 architecture spine
**When** M7 begins implementation
**Then** Athena introduces a dedicated `kernel/projection-model` boundary for projection-facing view definitions, projection elements, stable references, and inspectable projection payloads
**And** those contracts remain renderer-neutral rather than tied to Theia, GLSP, or any specific canvas library

**Given** projection contracts are reviewed against existing kernel ownership
**When** dependency direction is checked
**Then** `projection-model` consumes engineering, layout, and geometry meaning without redefining semantic identity
**And** view definitions and renderer assets remain downstream projection concerns rather than semantic roots

### Story 1.2: Materialize Deterministic Projection State From Engineering, Layout, And Geometry Inputs

As an engineer,
I want Athena to derive projection state deterministically from canonical inputs,
So that the same upstream semantic state always yields the same graphical projection state.

**Acceptance Criteria:**

**Given** canonical engineering identities, relationships, layout metadata, and geometry metadata
**When** Athena builds a projection model
**Then** it produces the same projection output for the same upstream inputs
**And** projection payloads remain inspectable for development and architecture debugging

**Given** renderer-facing structure is reviewed
**When** model ownership is checked
**Then** projection elements carry stable canonical references plus view-scoped projection identifiers
**And** renderer asset references stay separate from engineering identity

### Story 1.3: Host Runtime-Owned Projection Sessions With Deterministic Invalidation And Refresh

As a platform engineer,
I want runtime-owned projection sessions to control active graphical state,
So that stale or conflicting projection state is rebuilt deterministically from canonical inputs instead of surviving privately in the frontend.

**Acceptance Criteria:**

**Given** an active repository session and one or more graphical views
**When** Athena opens graphical projection state
**Then** runtime owns active `ProjectionSession` state beside the existing repository and semantic SCM context
**And** frontend workbench state remains disposable downstream state

**Given** upstream semantic, repository, or semantic SCM state changes
**When** the active projection session refreshes
**Then** stale projection state is invalidated and rebuilt deterministically from the current canonical inputs
**And** unapproved frontend-local state does not persist as authority

### Story 1.4: Expose Typed Projection Queries And Governed Commands Through `ide/lsp`

As an IDE engineer,
I want Athena to expose projection state through typed queries and an explicit governed command set,
So that graphical clients can inspect and request allowed actions without gaining ownership of semantic or projection truth.

**Acceptance Criteria:**

**Given** runtime-owned projection sessions exist
**When** the IDE requests graphical state
**Then** `ide/lsp` exposes typed projection queries and deterministic projection updates over Athena-owned protocol seams
**And** the transport remains inspectable and stable enough for later richer graphical milestones

**Given** graphical interaction needs to request state changes
**When** command surfaces are reviewed
**Then** only an explicit allowlist of governed commands is exposed
**And** generic runtime control tunnels or graph-framework-owned authority are not introduced

## Epic 2: Deliver The First Graphical Workbench Surface

Engineers can open a real graphical Athena view in the existing workbench, inspect engineering relationships visually, and move coherently between source, inspection, semantic SCM, and graphical context.

### Story 2.1: Introduce The Graph Adapter Boundary Under `integrations/graph-*`

As a platform engineer,
I want the first graphical framework integration to live behind a dedicated adapter seam,
So that protocol and rendering mechanics stay isolated from Athena semantic and projection contracts.

**Acceptance Criteria:**

**Given** Athena is ready to connect the product shell to a graphical client
**When** the first graph integration is introduced
**Then** framework-specific protocol and rendering code lives under `integrations/graph-*`
**And** that integration remains translation-only rather than semantic or projection authority

**Given** adapter behavior is reviewed
**When** ownership boundaries are checked
**Then** the adapter may translate protocol and rendering mechanics only
**And** it may not synthesize engineering truth, redefine semantic identity, or persist local state as authority

### Story 2.2: Surface A Graphical Athena View Inside The Existing Workbench

As an engineer,
I want Athena to open a real graphical view inside the current desktop workbench,
So that graphical projection becomes a product capability rather than an isolated demo.

**Acceptance Criteria:**

**Given** the current Athena Theia product shell
**When** users open a supported graphical projection
**Then** Athena presents that graphical view through existing additive product seams rather than a shell rewrite
**And** graphical views coexist with source, semantic inspection, repository, and semantic SCM surfaces

**Given** graphical panel lifecycle is reviewed
**When** workbench ownership is checked
**Then** Theia frontend and backend host panel lifecycle and presentation only
**And** runtime and `ide/lsp` remain the sole semantic and projection authorities

### Story 2.3: Synchronize Graphical Selection With Source, Semantic Inspection, And Semantic SCM Context

As an engineer or reviewer,
I want graphical selections to stay coherent with textual and semantic context,
So that I can move between source, inspection, review, and graphical understanding without semantic drift.

**Acceptance Criteria:**

**Given** source, semantic inspection, semantic SCM, and a graphical view are open
**When** users select or focus supported graphical elements
**Then** Athena synchronizes that selection with the relevant canonical source and semantic context
**And** the same repository and semantic state produce the same synchronized inspection outcome

**Given** later visual review and history work are planned
**When** synchronization boundaries are reviewed
**Then** existing M6 semantic identity and history vocabulary are reused
**And** the graphical path does not invent renderer-specific change identities

### Story 2.4: Keep Graphical Interaction Inspect-First And Transient By Default

As a platform engineer,
I want M7 graphical interaction to stay inspect-first,
So that the first visual milestone proves useful interaction without widening into unrestricted graphical authoring.

**Acceptance Criteria:**

**Given** a graphical workbench view is open
**When** users select, focus, reveal, or navigate supported graphical elements
**Then** those interactions behave as downstream UI actions over runtime-owned projection state
**And** they do not mutate semantic or persisted projection state privately

**Given** unapproved frontend interaction occurs or a graphical client offers richer default behavior
**When** Athena refreshes projection state
**Then** that unapproved state either snaps back or is discarded on refresh
**And** any persisted projection change must route through governed runtime commands

## Epic 3: Prove The First Renderer And Lock The Technology Path

Architecture and product owners can validate a relationship-forward first renderer, domain projection contribution path, and the first graph technology fit without widening M7 into unrestricted editing.

### Story 3.1: Build The Relationship-Forward First Renderer Proof Over Canonical Object Identities

As an engineer,
I want Athena to render a relationship-forward engineering projection over canonical objects,
So that the first graphical proof demonstrates object-graph-first projection rather than notation-first drawing ownership.

**Acceptance Criteria:**

**Given** projection-model contracts and runtime-owned projection sessions exist
**When** Athena renders the first supported graphical proof
**Then** the result centers on engineering objects and relationships over stable canonical identities
**And** the proof remains useful without notation-specific symbol or asset packs

**Given** the first renderer is reviewed against the M7 milestone intent
**When** its behavior is inspected
**Then** it proves renderer-target delivery over canonical objects rather than drawing-file ownership
**And** it keeps later notation-specific projection possible without making notation the semantic center

**Given** engineers are using an authored `.athena` document such as `factory-line.athena`
**When** they inspect the first serious renderer proof in the Athena workbench
**Then** the graph surface is presented as the primary work area in a docked split layout with the source kept visible as secondary context
**And** the presentation favors dense, professional engineering-workbench information layout rather than spacious dashboard-like text panels

**Given** Athena is proving the first serious renderer surface rather than a report view
**When** engineers navigate the diagram
**Then** the surface behaves like an infinite canvas with pan/zoom-oriented navigation and a graph-first viewport
**And** the implementation avoids presenting the renderer as a small bounded SVG preview card

### Story 3.2: Route Electrical View Definitions And Renderer Mappings Through `extensions/domain-electrical`

As a domain platform engineer,
I want electrical projection contributions to enter through the extension layer,
So that M7 can prove the first domain projection path without hard-coding domain view logic into kernel contracts.

**Acceptance Criteria:**

**Given** the existing `extensions/domain-electrical` proof domain
**When** Athena adds the first domain-specific projection contributions
**Then** electrical view definitions and renderer mappings enter through extension-owned contribution paths
**And** kernel projection contracts remain renderer-neutral and domain-neutral

**Given** later domains are considered
**When** extension boundaries are reviewed
**Then** the same contribution model remains usable for future mechanical, process, or other view families
**And** M7 does not require kernel contract rewrites to support them

### Story 3.3: Validate The First Graphical Technology Path And Publish The M7 Proof Corpus

As an architecture owner,
I want Athena to validate the first graphical technology path and publish proof artifacts,
So that M7 ends with a credible implementation direction and a repeatable proof corpus instead of an unresolved framework debate.

**Acceptance Criteria:**

**Given** the first graph adapter and renderer proof are implemented
**When** Athena evaluates the chosen graphical technology path
**Then** it compares the implementation against M7 constraints such as JVM-first authority, Theia product-shell fit, translation-only adapters, inspectability, and deterministic refresh
**And** the decision is recorded clearly enough for later milestone planning

**Given** M7 is reviewed as a milestone proof
**When** users inspect the proof corpus
**Then** Athena provides examples, runnable verification, or equivalent milestone artifacts that demonstrate the relationship-forward renderer and product integration path
**And** the milestone remains focused on graphical projection rather than broad UX-polish or unrestricted editing scope
