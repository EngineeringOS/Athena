# Athena - Epic Breakdown

## Overview

This document provides the epic and story breakdown for Athena M15, decomposing guided semantic authoring into governed, implementable work above the completed M14 component-knowledge foundation.

M15 is a product-viability milestone, not a palette-only UI milestone and not a second mutation-system milestone.

Its job is to prove that mainstream engineers can create and modify governed engineering intent through:

- component panel or search driven placement
- inspector-driven property editing
- port-aware connection authoring
- shared reveal across source, graph, inspector, and semantic SCM
- preview-first mutation acceptance

while preserving:

- M8 as the only mutation authority
- M14 as the source of authorable component knowledge
- canonical `Engineering IR` as the only authored truth
- DSL as canonical serialization rather than the required primary UI

## Requirements Inventory

### Functional Requirements

FR-1: Athena can publish authoring intent as a first-class platform contract.
FR-2: Athena can preserve M8 as the only mutation authority for guided authoring.
FR-3: Athena can publish a governed component palette from active component-knowledge packs.
FR-4: Athena can insert components through guided placement intents.
FR-5: Athena can publish a component inspector from canonical semantic identity.
FR-6: Athena can edit component properties through governed update intents.
FR-7: Athena can start connection authoring from semantic ports.
FR-8: Athena can filter allowed targets by port meaning rather than graph shape.
FR-9: Athena can create connections through governed connect intents.
FR-10: Athena can keep source, graph, and guided authoring in sync.
FR-11: Athena can preserve one identity everywhere.
FR-12: Athena can preview guided mutations before acceptance.
FR-13: Athena can commit accepted guided mutations into canonical state.

### NonFunctional Requirements

NFR-1: Guided authoring introduces no second write path outside M8.
NFR-2: Workbench surfaces remain consumers of platform-owned authoring contracts.
NFR-3: Available components, parts, ports, and traits derive from active M14 component knowledge rather than frontend hardcoding.
NFR-4: Canonical semantic identity remains stronger than graph element ids, widget ids, or presentation occurrence ids.
NFR-5: Preview and approval remain inspectable and deterministic.
NFR-6: The first proof stays narrow, Siemens-first, and electrical only.
NFR-7: M15 excludes full routing, macro authoring, multi-vendor breadth, AI-primary generation, and full EPLAN parity.

### Additional Requirements

- M15 builds on completed M8 mutation authority and may not bypass it from any workbench surface.
- M15 builds on completed M14 component knowledge and must consume active authorable component definitions, semantic ports, and vendor implementations from that layer.
- M15 should keep the initial guided authoring proof inside the current Theia and GLSP workbench stack rather than inventing a second desktop shell.
- The first proof stays narrow around PLC CPU, power supply, motor, contactor, and one compatible connection slice.
- Guided authoring should remain review-first and align with existing M6 and M8 preview or commit semantics where possible.

### UX Design Requirements

- The first workbench proof should expose an Athena component panel in the left sidebar.
- The first workbench proof should expose an Athena inspector in the right sidebar.
- The graph workbench should host the first port-aware connect gesture.
- The workbench should not require the user to edit raw DSL to complete the proof flow.

### FR Coverage Map

FR-1: Epic 1 - authoring model foundation
FR-2: Epic 1 and Epic 5 - mutation-authority preservation and preview-first commit
FR-3: Epic 2 - governed component panel and available-component publishing
FR-4: Epic 2 - guided component placement
FR-5: Epic 3 - inspector snapshot and canonical identity binding
FR-6: Epic 3 - governed inspector editing
FR-7: Epic 4 - graph-originated semantic port connection start
FR-8: Epic 4 - allowed-target filtering
FR-9: Epic 4 - governed connect intent and connection persistence
FR-10: Epic 2, Epic 3, and Epic 5 - cross-surface synchronization
FR-11: Epic 5 - one identity everywhere and shared reveal
FR-12: Epic 1 and Epic 5 - preview contracts and guided mutation review
FR-13: Epic 5 - commit accepted guided mutations into canonical state

## Epic 1: Authoring Intent And Runtime Foundation

Platform engineers can define guided authoring contracts and preview semantics so every M15 workbench action routes through one platform-owned authoring layer above M8 instead of through frontend shortcuts.

### Story 1.1: Publish The Core Authoring Intent Contract

As a platform engineer,  
I want Athena to define typed guided-authoring intents,  
So that palette, inspector, and graph gestures emit platform-owned requests rather than frontend-local mutation logic.

**Implements:** FR-1, FR-2

**Acceptance Criteria:**

**Given** M15 introduces guided authoring  
**When** the contract is reviewed  
**Then** Athena defines typed authoring intents for at least component creation, property update, connection creation, and reveal

**Given** workbench actions are compared across panel and graph surfaces  
**When** authoring responsibilities are reviewed  
**Then** the shared contract belongs to platform modules rather than to Theia widget code or GLSP-local behavior

### Story 1.2: Define Preview And Acceptance Contracts For Guided Authoring

As an architecture owner,  
I want Athena to define previewable authoring outcomes before commit,  
So that guided authoring stays review-first and inspectable.

**Implements:** FR-12, FR-13

**Acceptance Criteria:**

**Given** one guided authoring action is requested  
**When** the runtime expands it  
**Then** Athena can publish a preview of the semantic consequences before acceptance

**Given** a preview is accepted or rejected  
**When** the flow is completed  
**Then** acceptance hands off to canonical mutation while rejection leaves canonical state unchanged

### Story 1.3: Publish Shared Authoring Runtime And Transport Seams

As a runtime engineer,  
I want guided authoring to flow through shared runtime and transport seams,  
So that M15 workbench surfaces stay thin consumers of one authoring service.

**Implements:** FR-1, FR-2

**Acceptance Criteria:**

**Given** the first M15 workbench proof is implemented  
**When** the service boundaries are reviewed  
**Then** authoring operations are exposed through runtime-owned services and Athena-owned transport rather than through ad hoc frontend mutation paths

**Given** future forms or AI surfaces are considered  
**When** the authoring service contract is inspected  
**Then** the contract can be reused without depending on one specific workbench widget

## Epic 2: Guided Component Placement

Engineers can browse governed available components and insert one component instance through guided placement so Athena becomes a creation surface rather than only an inspection surface.

### Story 2.1: Publish The First Governed Component Panel

As an engineer,  
I want Athena to show available components in a dedicated component panel,  
So that I can author from governed component knowledge without writing DSL directly.

**Implements:** FR-3

**Acceptance Criteria:**

**Given** one governed repository is open  
**When** the Athena component panel is displayed  
**Then** available components are loaded from active component-knowledge packs instead of from hardcoded frontend lists

**Given** the first proof slice is electrical only  
**When** categories are reviewed  
**Then** the panel can group the first proof set by narrow categories such as PLC, power supply, motor, and contactor

### Story 2.2: Insert Components Through Guided Placement Intents

As an engineer,  
I want to insert one component instance through the workbench,  
So that Athena creates governed engineering intent rather than a temporary graph shape.

**Implements:** FR-4

**Acceptance Criteria:**

**Given** a user double-clicks, commands, or drags one available component  
**When** insertion is requested  
**Then** Athena emits a guided placement intent rather than directly creating a graph node

**Given** the insertion preview is accepted  
**When** canonical state is rebuilt  
**Then** the project contains one new component instance in source and graph

### Story 2.3: Keep Placement Synchronized Across Source And Graph

As a product owner,  
I want guided placement to refresh all relevant workbench surfaces,  
So that component insertion feels like one coherent authoring action.

**Implements:** FR-10

**Acceptance Criteria:**

**Given** a component insertion is accepted  
**When** the runtime completes canonical rebuild  
**Then** source, graph, diagnostics, and selection-aware panels refresh coherently

**Given** the same inserted subject is revealed  
**When** the user moves between workbench surfaces  
**Then** the same canonical semantic identity remains selected and traceable

## Epic 3: Inspector Editing

Engineers can inspect and update one selected component through canonical semantic identity so vendor implementation and authorable properties are no longer trapped in raw DSL.

### Story 3.1: Publish A Canonical Inspector Snapshot

As an engineer,  
I want Athena to show component details in an inspector,  
So that I can understand and edit the selected component through one governed surface.

**Implements:** FR-5

**Acceptance Criteria:**

**Given** a component is selected in the graph or source  
**When** the inspector is opened  
**Then** the inspector shows component name, engineering concept, vendor implementation, semantic ports, and minimal physical traits

**Given** the selected subject is compared across graph and inspector  
**When** identity is reviewed  
**Then** the inspector binds to canonical semantic identity rather than to frontend-local node ids

### Story 3.2: Update Editable Properties Through Governed Intents

As an engineer,  
I want to edit selected component properties through the inspector,  
So that I can change tag, description, or implementation without editing DSL directly.

**Implements:** FR-6

**Acceptance Criteria:**

**Given** the first M15 proof supports narrow editable properties  
**When** the inspector edits one selected component  
**Then** Athena can update at least tag or name, description, vendor implementation choice, and custom labels through a governed update intent

**Given** the preview is accepted  
**When** canonical state is rebuilt  
**Then** source, graph, diagnostics, and inspector state refresh coherently

### Story 3.3: Keep Implementation, Port, And Trait Details Coherent

As an architecture owner,  
I want inspector detail rendering to stay consistent with active component knowledge,  
So that the workbench does not become a second catalog model.

**Implements:** FR-5, FR-10

**Acceptance Criteria:**

**Given** the inspector shows part options, ports, and physical traits  
**When** those details are reviewed  
**Then** they derive from active component knowledge rather than from duplicated frontend truth

**Given** the selected implementation changes  
**When** the inspector refreshes  
**Then** the shown ports and minimal physical traits remain consistent with the resolved component knowledge

## Epic 4: Port-Aware Connection Authoring

Engineers can create one compatible connection from semantic ports through guided filtering and governed mutation so graph interaction stays semantic instead of generic edge drawing.

### Story 4.1: Publish Allowed Connection Targets From Semantic Ports

As an engineer,  
I want Athena to show only compatible connection targets,  
So that I can connect components through port meaning rather than by guessing graph shapes.

**Implements:** FR-7, FR-8

**Acceptance Criteria:**

**Given** a user starts a connection from one semantic port  
**When** allowed targets are requested  
**Then** Athena returns targets filtered by stable direction, signal family, and narrow protocol-bearing metadata

**Given** incompatible targets exist on the same canvas  
**When** the connect flow is active  
**Then** those targets are not presented as valid completions

### Story 4.2: Create Connections Through Governed Connect Intents

As an engineer,  
I want Athena to create one connection through a guided flow,  
So that graph interaction results in canonical engineering intent rather than a temporary line.

**Implements:** FR-9

**Acceptance Criteria:**

**Given** a compatible target is chosen  
**When** the connection action is previewed and accepted  
**Then** Athena creates one canonical connection through governed mutation instead of direct graph-edge creation

**Given** a connection was accepted  
**When** canonical state rebuild completes  
**Then** the connection appears in source, graph, and relevant inspector state

### Story 4.3: Reflect Connection State Across Graph And Inspector

As a product owner,  
I want the workbench to show connection state coherently after guided connect,  
So that the connection flow feels like one semantic operation.

**Implements:** FR-9

**Acceptance Criteria:**

**Given** one compatible connection exists  
**When** the connected ports are inspected  
**Then** the inspector can show connection state for both ends

**Given** the connection is selected or revealed  
**When** graph and inspector are compared  
**Then** both surfaces remain anchored to canonical connection identity and endpoints

## Epic 5: Unified Reveal, Review, And Proof

Product and platform owners can prove three-way synchronization, one identity everywhere, and preview-first guided commit so M15 demonstrates real product usability rather than isolated UI widgets.

### Story 5.1: Preserve One Identity Everywhere

As an engineer,  
I want to reveal the same subject across source, graph, inspector, and semantic SCM,  
So that Athena feels like one governed product rather than disconnected tools.

**Implements:** FR-10, FR-11

**Acceptance Criteria:**

**Given** a component or connection is selected from any M15 surface  
**When** reveal is requested elsewhere  
**Then** source, graph, inspector, and semantic SCM resolve to the same canonical semantic identity

**Given** presentation or graph ids differ from semantic ids  
**When** reveal behavior is reviewed  
**Then** canonical semantic identity remains the anchor and downstream ids remain secondary

### Story 5.2: Reuse Review-First Mutation Preview For Guided Authoring

As a product owner,  
I want guided authoring to remain preview-first before commit,  
So that engineers can inspect what will change before Athena persists it.

**Implements:** FR-12, FR-13

**Acceptance Criteria:**

**Given** a create, update, or connect action is requested  
**When** preview is shown  
**Then** the preview can describe what semantic subjects or properties will be added or changed

**Given** preview semantics are compared with M6 and M8 review patterns  
**When** the product flow is reviewed  
**Then** guided authoring aligns with the existing review-first direction rather than introducing opaque direct commit

### Story 5.3: Publish The First Guided Authoring Proof Corpus And Verification Path

As a platform owner,  
I want Athena to publish one narrow repository-backed proof and deterministic verification path for guided authoring,  
So that M15 closes with real product evidence rather than UI-only demos.

**Implements:** FR-10, FR-13

**Acceptance Criteria:**

**Given** the first M15 proof corpus is reviewed  
**When** the scenario is inspected  
**Then** it demonstrates open repository -> insert PLC -> edit tag -> add supply -> connect compatible ports -> preview -> approve -> verify source and graph coherence

**Given** the milestone closes  
**When** verification evidence is recorded  
**Then** the proof path documents the exact targeted runtime, LSP, frontend, and repository-backed validation steps used to support M15 claims
