# Athena - Epic Breakdown

## Overview

This document provides the epic and story breakdown for Athena M17, decomposing parser architecture hardening into implementable work above the completed M16 semantic reuse foundation.

M17 is a language architecture milestone, not a broad syntax-expansion milestone.

Its job is to prove that Athena can deliver durable language infrastructure through:

- a preserved authored AST boundary
- `ANTLR4` as the compiler and LSP parser
- `Tree-sitter` as the IDE syntax parser
- compiler/LSP continuity on the current supported syntax subset
- parser parity and malformed-source proof coverage

while preserving:

- `Engineering IR` as the only canonical semantic authority
- `ide/lsp` as the only semantic entry point for IDE meaning
- direct DSL as canonical serialization rather than the default human interface
- existing lowering, runtime, and downstream semantic ownership boundaries

## Requirements Inventory

### Functional Requirements

FR-1: Athena can preserve one explicit authored AST boundary.
FR-2: Athena can publish a dual-parser responsibility model.
FR-3: Athena can parse supported Athena syntax through `ANTLR4`.
FR-4: Athena can preserve lowering through authored AST instead of parse trees.
FR-5: Athena can publish a Tree-sitter grammar for Athena source.
FR-6: Athena can use Tree-sitter for syntax UX rather than semantic truth.
FR-7: Athena can keep compiler and LSP semantics on the compiler parser path.
FR-8: Athena can preserve useful failure behavior on invalid source.
FR-9: Athena can define a durable landing zone for future syntax.
FR-10: Athena can publish a verification corpus for parser parity and IDE behavior.

### NonFunctional Requirements

NFR-1: `Engineering IR` remains canonical engineering truth.
NFR-2: Generated parser-tree types do not become public semantic contracts.
NFR-3: Source spans and diagnostics remain inspectable across parser migration.
NFR-4: Editor syntax parsing remains error tolerant and low latency.
NFR-5: The M17 architecture must make future syntax additions cheaper, not harder.

### Additional Requirements

- M17 builds on the existing `:kernel:language` syntax-only AST boundary and should harden it rather than discard it.
- M17 builds on the current lowering path and must keep lowering operating on Athena-owned AST contracts.
- M17 must not widen into final `import` semantics, final package-aware authored semantics, or full expression-language delivery.
- M17 must keep Tree-sitter out of canonical semantic truth and out of `Engineering IR` derivation.
- M17 must keep semantic diagnostics on the compiler/LSP path.
- M17 should prefer real checked-in proof inputs over toy parser-only demos.
- M17 should include malformed or incomplete source proofs in addition to valid-source parity proofs.
- M17 should preserve source-edit, reveal, and document-symbol usefulness where current language services depend on the syntax model.

### UX Design Requirements

- The first editor proof should expose at least one serious Tree-sitter-backed syntax UX capability such as highlighting, folding, or outline.
- The editor should remain usable on incomplete or malformed source.
- The product should continue to surface compiler/LSP diagnostics as the semantic truth path rather than inventing a new frontend-only diagnostic model.

### FR Coverage Map

FR-1: Epic 1 and Epic 2 - authored AST boundary preservation and parser adaptation
FR-2: Epic 1 and Epic 3 - dual-parser model and editor/parser split
FR-3: Epic 2 - ANTLR compiler parsing of the supported syntax subset
FR-4: Epic 1 and Epic 2 - lowering through authored AST only
FR-5: Epic 3 - Tree-sitter grammar publication
FR-6: Epic 3 and Epic 4 - syntax UX only, semantic truth stays elsewhere
FR-7: Epic 4 - compiler/LSP semantic continuity
FR-8: Epic 2, Epic 3, and Epic 5 - invalid-source diagnostics and usable editor behavior
FR-9: Epic 1 and Epic 5 - durable landing zone for future syntax
FR-10: Epic 5 - parser parity and IDE verification corpus

## Epic 1: Authored AST Boundary And Syntax Contract Hardening

Platform engineers can harden Athena's syntax boundary as a stable authored AST contract so parser implementation can evolve without leaking into lowering, runtime, or downstream semantic meaning.

### Story 1.1: Freeze The Public Authored Syntax Contract

As a platform engineer,
I want Athena to define one stable authored AST and parse-result contract,
So that compiler parsing can change without forcing downstream semantic consumers to depend on parser-generator internals.

**Implements:** FR-1, FR-2

**Acceptance Criteria:**

**Given** M17 hardens the language substrate
**When** the syntax boundary is reviewed
**Then** Athena publishes AST, parse result, span, and syntax diagnostic contracts as Athena-owned types rather than generated parser-tree types

**Given** downstream consumers are inspected
**When** lowering, runtime, IDE, and tests are reviewed
**Then** they depend on Athena syntax contracts rather than on ANTLR or Tree-sitter implementation classes

### Story 1.2: Isolate Parser Implementation Behind The Language Facade

As a platform engineer,
I want parser implementation details isolated behind `:kernel:language`,
So that parser migration does not become a public architecture leak.

**Implements:** FR-1, FR-2, FR-4

**Acceptance Criteria:**

**Given** compiler parsing internals are implemented
**When** the language boundary is inspected
**Then** generated parser artifacts remain implementation detail behind Athena-owned parse APIs

**Given** future parser changes are considered
**When** the public syntax seam is reviewed
**Then** the facade is stable enough that downstream modules do not need parser-generator-specific imports

### Story 1.3: Preserve AST Extensibility For Future Syntax

As a language architect,
I want the authored AST boundary to leave room for future constructs such as imports,
So that M17 becomes a foundation for language growth instead of a one-off migration.

**Implements:** FR-1, FR-9

**Acceptance Criteria:**

**Given** future authored constructs are considered
**When** the AST and parser adaptation boundary are reviewed
**Then** the design leaves a deliberate landing zone for future syntax without forcing direct parse-tree-driven lowering

**Given** M17 scope is inspected
**When** milestone commitments are reviewed
**Then** the architecture prepares for future syntax growth
**And** it does not claim to finalize that future syntax in M17

## Epic 2: ANTLR4 Compiler Parser Migration

Compiler engineers can parse the current supported Athena syntax through `ANTLR4` and adapt it into the authored AST so future language growth no longer depends on the handwritten parser remaining small forever.

### Story 2.1: Publish The ANTLR4 Grammar For The Current Supported Syntax Subset

As a compiler engineer,
I want Athena to define an `ANTLR4` grammar for the current authored syntax subset,
So that compiler parsing moves onto a durable parser technology without widening syntax scope prematurely.

**Implements:** FR-3

**Acceptance Criteria:**

**Given** the current Athena source subset is reviewed
**When** the compiler grammar is implemented
**Then** it supports `system`, `device`, `port`, `connect`, qualified names, string literals, and property assignments

**Given** M17 remains parity-first
**When** grammar scope is inspected
**Then** the first proof targets today's supported syntax rather than full future-language breadth

### Story 2.2: Adapt ANTLR Parse Trees Into The Authored AST

As a compiler engineer,
I want Athena to adapt ANTLR parser output into the existing authored AST seam,
So that lowering remains stable and parser internals remain isolated.

**Implements:** FR-1, FR-4

**Acceptance Criteria:**

**Given** ANTLR parser output exists
**When** the compiler prepares syntax results
**Then** Athena adapts the parse tree into authored AST nodes rather than lowering parse trees directly

**Given** lowering is reviewed
**When** the dependency boundary is inspected
**Then** `Engineering IR` derivation still consumes authored AST only

### Story 2.3: Preserve Provenance And Failure Quality On The ANTLR Path

As a compiler engineer,
I want the ANTLR-backed parser to preserve spans and useful diagnostics,
So that parser migration does not regress source edits, reveal, or syntax failure inspection.

**Implements:** FR-3, FR-8

**Acceptance Criteria:**

**Given** valid source is parsed
**When** syntax nodes are inspected
**Then** authored AST nodes preserve usable file, line, column, and span provenance

**Given** malformed source is parsed
**When** parsing fails
**Then** Athena emits inspectable syntax diagnostics rather than opaque parser crashes or positionless errors

## Epic 3: Tree-sitter IDE Syntax Path

IDE engineers can use Tree-sitter for syntax UX on the supported Athena syntax subset so the editor becomes more incremental and error tolerant without becoming a second semantic compiler.

### Story 3.1: Publish The Tree-sitter Grammar For Athena Syntax UX

As an IDE engineer,
I want Athena to publish a Tree-sitter grammar for the current supported source subset,
So that editor syntax behavior has an incremental and error-tolerant parser foundation.

**Implements:** FR-5

**Acceptance Criteria:**

**Given** the current Athena source subset is reviewed
**When** the Tree-sitter grammar is implemented
**Then** it can recognize the current authored forms used in checked-in proof inputs

**Given** the editor parser boundary is reviewed
**When** grammar ownership is inspected
**Then** the Tree-sitter grammar remains scoped to syntax structure rather than semantic compilation

### Story 3.2: Integrate Tree-sitter Into One Supported Theia Syntax UX Path

As an IDE engineer,
I want Athena to use Tree-sitter for one real syntax UX path in Theia,
So that M17 proves editor value instead of shipping a detached grammar only.

**Implements:** FR-5, FR-6

**Acceptance Criteria:**

**Given** the first M17 IDE proof is implemented
**When** the editor is used on Athena source
**Then** at least one serious syntax UX capability such as highlighting, folding, or outline is backed by Tree-sitter

**Given** the integration path is reviewed
**When** ownership is inspected
**Then** the feature stays inside supported `ide/theia-frontend` seams and does not create a second semantic parser path

### Story 3.3: Prove Incomplete-Source Editor Tolerance

As an IDE engineer,
I want Tree-sitter-backed syntax UX to stay usable on incomplete source,
So that the editor remains responsive while the compiler path still owns semantic failure.

**Implements:** FR-6, FR-8

**Acceptance Criteria:**

**Given** a source file is incomplete or malformed
**When** the editor updates
**Then** Tree-sitter still yields a usable syntax tree for the chosen syntax UX path

**Given** the same malformed source is reviewed semantically
**When** diagnostics are requested
**Then** semantic failure still comes from the compiler/LSP path rather than from Tree-sitter pretending to be semantic truth

## Epic 4: Compiler And LSP Language-Service Continuity

Compiler and IDE engineers can preserve semantic diagnostics and core language-service behavior on the compiler parser path so parser migration does not fracture product semantics.

### Story 4.1: Keep Semantic Diagnostics On The Compiler Parser Path

As a platform engineer,
I want Athena to keep LSP syntax and semantic diagnostics on the compiler path,
So that Tree-sitter integration never becomes a second semantic truth source.

**Implements:** FR-6, FR-7

**Acceptance Criteria:**

**Given** the IDE requests diagnostics
**When** parser migration is complete
**Then** `ide/lsp` still derives diagnostics from compiler-owned parsing and later compiler/runtime stages

**Given** Tree-sitter is active in the editor
**When** semantic meaning is inspected
**Then** Tree-sitter does not become the source of semantic diagnostics or engineering truth

### Story 4.2: Preserve Source Navigation And Symbol Utility Across Parser Migration

As an IDE engineer,
I want current AST-dependent editor and LSP utilities to remain useful after parser migration,
So that M17 does not regress existing language-service value while hardening the parser architecture.

**Implements:** FR-7, FR-8

**Acceptance Criteria:**

**Given** existing source-oriented language services are reviewed
**When** the ANTLR-backed AST path replaces the handwritten parser path
**Then** reveal, source-edit, and document-symbol-style utilities continue to operate on Athena-owned syntax contracts

**Given** syntax provenance is inspected
**When** those services resolve spans and declarations
**Then** their behavior remains grounded in preserved authored-AST spans rather than parser-generator internals

### Story 4.3: Preserve Compiler Output Continuity On Supported Source

As a compiler engineer,
I want compiler outputs on supported source to remain semantically continuous across parser migration,
So that M17 proves architecture hardening rather than accidental semantic drift.

**Implements:** FR-4, FR-7

**Acceptance Criteria:**

**Given** one supported Athena source input
**When** it is compiled through the M17 parser path
**Then** lowering continues to produce the expected canonical `Engineering IR` shape for the supported subset

**Given** parser parity is reviewed
**When** current valid proof inputs are compared before and after migration
**Then** semantic continuity is preserved for the supported grammar surface
