# Athena - Epic Breakdown

## Overview

This document provides the epic and story breakdown for Athena M14, decomposing the new component-knowledge foundation into governed, implementable work above the completed M13 presentation-language milestone.

M14 is a kernel-foundation milestone, not a graphical-authoring milestone and not a behavior-model milestone.

Its job is to establish deterministic component knowledge resolution through:

- engineering concept contracts
- semantic port contracts
- minimal physical-trait contracts
- vendor implementation mappings
- package-governed knowledge-pack loading

while preserving:

- M8 as the only mutation authority
- M9 as the knowledge-runtime layer above resolved component knowledge
- M13 as the downstream presentation-language layer

## Requirements Inventory

### Functional Requirements

FR-1: Athena can resolve authored component references into governed engineering concepts.
FR-2: Athena can publish component knowledge through dedicated kernel contracts.
FR-3: Athena can define typed semantic port contracts.
FR-4: Athena can keep rich compatibility logic outside the port contract and downstream in M9.
FR-5: Athena can publish minimal physical-trait contracts.
FR-6: Athena can model vendor parts as implementations of engineering concepts.
FR-7: Athena can load component-knowledge packs deterministically through existing package governance.
FR-8: Athena can surface conflicting definitions explicitly.
FR-9: Athena can feed resolved component knowledge into M9 and M13 downstream consumers.
FR-10: Athena can preserve one mutation authority and the multi-surface product direction.

### NonFunctional Requirements

NFR-1: Canonical authored engineering meaning remains in `Engineering IR`.
NFR-2: M14 introduces no second mutation or review authority.
NFR-3: Knowledge-pack loading remains deterministic and constrained by the existing package graph and `athena.lock`.
NFR-4: Engineering concepts remain vendor-neutral; vendor parts remain implementations.
NFR-5: Semantic ports remain kernel knowledge contracts and not renderer-local affordances.
NFR-6: Physical traits remain minimal and do not become a geometry engine.
NFR-7: M14 output remains read-only resolved knowledge consumed by later layers.
NFR-8: M14 excludes behavior model, simulation, broad standards packs, and broad catalog parity.

### Additional Requirements

- M14 builds on completed M5 package governance and must reuse that deterministic package foundation instead of introducing ad hoc pack discovery.
- M14 builds on completed M8 mutation authority and may not open a new write path.
- M14 builds on completed M9 knowledge runtime and must position resolved component knowledge as later M9 input, not as a replacement for M9.
- M14 builds on completed M13 presentation language and must position resolved component knowledge as later projection or presentation input, not as a replacement for M13.
- The first proof stays electrical-only.
- The first proof vendor stays Siemens-only.
- The first proof component families remain narrow: PLC CPU, I/O, contactor, relay, motor, 24V power supply, breaker, and cable.
- Direct DSL authoring remains valid, but M14 must preserve the product rule that DSL is canonical serialization rather than the required mainstream user interface.

### UX Design Requirements

None included for M14 core.

M14 may enable later graphical, form-based, template-based, and AI-assisted authoring, but it does not itself implement the mainstream graphical authoring milestone.

### FR Coverage Map

FR-1: Epic 1 and Epic 3 - engineering concept contracts and deterministic resolution
FR-2: Epic 1 and Epic 2 - dedicated kernel contracts
FR-3: Epic 2 - semantic port contracts
FR-4: Epic 2 and Epic 4 - narrow port contracts plus downstream M9 integration
FR-5: Epic 2 - minimal physical-trait contracts
FR-6: Epic 1 and Epic 3 - part model and vendor implementation mappings
FR-7: Epic 3 - deterministic knowledge-pack loading
FR-8: Epic 3 and Epic 5 - explicit conflict handling and proof diagnostics
FR-9: Epic 4 - downstream integration into M9 and M13 consumers
FR-10: Epic 4 and Epic 5 - preserved mutation authority and product-position proof

## Epic 1: Component And Part Model Foundation

Platform engineers can define vendor-neutral engineering concepts and vendor implementation mappings so authored component references stop being treated as thin labels or vendor-shaped semantics.

### Story 1.1: Define The Core Engineering Concept Contract

As a platform engineer,  
I want Athena to define a vendor-neutral engineering concept contract,  
So that authored component references resolve into stable engineering meaning instead of only vendor ids or loose kind strings.

**Implements:** FR-1, FR-2

**Acceptance Criteria:**

**Given** M14 introduces component knowledge  
**When** the core contract is reviewed  
**Then** engineering concept identity is defined in a dedicated kernel module rather than inside compiler-local helper structures  
**And** the contract is vendor-neutral and suitable for future domain or vendor packs

**Given** one authored component reference is resolved  
**When** Athena publishes the resolved result  
**Then** the result includes both canonical authored semantic subject identity and resolved engineering concept identity

### Story 1.2: Define The Vendor Implementation Mapping Contract

As a platform engineer,  
I want Athena to model vendor parts as implementations of engineering concepts,  
So that vendor ids stop acting as the semantic type system.

**Implements:** FR-2, FR-6

**Acceptance Criteria:**

**Given** the first M14 part-model contract is reviewed  
**When** architecture owners inspect its responsibilities  
**Then** the contract separates engineering concept identity from vendor implementation identity

**Given** one concept has multiple possible realizations  
**When** vendor implementations are published  
**Then** the model can represent more than one implementation without redefining the concept itself

### Story 1.3: Publish The First Electrical Concept And Siemens Implementation Slice

As an electrical platform owner,  
I want Athena to ship the first narrow electrical concept and Siemens implementation slice,  
So that M14 proves the architecture over real component families instead of abstract placeholders.

**Implements:** FR-1, FR-6

**Acceptance Criteria:**

**Given** the first M14 proof pack is implemented  
**When** electrical concept definitions are reviewed  
**Then** the proof includes at least PLC CPU, contactor, relay, motor, and 24V power supply concepts

**Given** Siemens is the first vendor proof  
**When** implementation mappings are reviewed  
**Then** at least one Siemens implementation exists for each targeted proof family

## Epic 2: Semantic Port And Physical Trait Foundation

Platform engineers can define typed semantic ports and minimal physical traits so later knowledge, projection, and presentation layers consume reusable component meaning instead of ad hoc metadata.

### Story 2.1: Define Typed Semantic Port Contracts

As a platform engineer,  
I want Athena to define semantic port contracts with stable engineering meaning,  
So that ports stop being only labels and become reusable connection concepts.

**Implements:** FR-2, FR-3

**Acceptance Criteria:**

**Given** the first semantic port contract is reviewed  
**When** port responsibilities are inspected  
**Then** the contract can express stable role, direction, signal family, and optional protocol-bearing metadata

**Given** semantic ports are compared with renderer and frontend systems  
**When** ownership is reviewed  
**Then** the port contract remains kernel knowledge and does not depend on graph coordinates, shape ids, or widget state

### Story 2.2: Keep Compatibility Judgement Out Of The Port Contract

As an architecture owner,  
I want Athena to keep rich compatibility logic outside the port contract,  
So that M14 does not collapse into a second rule-engine milestone.

**Implements:** FR-4

**Acceptance Criteria:**

**Given** the connection-model contract is reviewed  
**When** M9 responsibilities are compared with M14 responsibilities  
**Then** semantic port contracts define type and role only  
**And** richer compatibility or sufficiency evaluation remains downstream M9 logic

### Story 2.3: Define Minimal Physical-Trait Contracts

As a platform engineer,  
I want Athena to define minimal physical-trait contracts,  
So that later layout, projection, and presentation layers can consume reusable physical meaning without making M14 a geometry milestone.

**Implements:** FR-2, FR-5

**Acceptance Criteria:**

**Given** the first physical-model contract is reviewed  
**When** its scope is inspected  
**Then** the model covers width, height, depth, mounting type, and basic installation markers only

**Given** physical traits are compared with layout or geometry layers  
**When** ownership is reviewed  
**Then** physical traits remain reusable component knowledge and do not replace layout or scene-calculation ownership

### Story 2.4: Publish The First Electrical Semantic Port And Physical-Trait Slice

As an electrical platform owner,  
I want Athena to ship the first electrical semantic port and physical-trait slice,  
So that the component knowledge proof includes reusable connection and mounting meaning.

**Implements:** FR-3, FR-5

**Acceptance Criteria:**

**Given** the first PLC CPU proof concept is resolved  
**When** the resolved knowledge is inspected  
**Then** semantic ports include at least `L+`, `M`, `PE`, and `MPI`

**Given** the first Siemens proof slice is resolved  
**When** physical traits are inspected  
**Then** at least one targeted proof component publishes minimal dimensions and mounting type

## Epic 3: Deterministic Knowledge-Pack Loading And Resolution

Compiler and platform engineers can load component-knowledge packs deterministically through the existing package graph and resolve component references with explicit conflict diagnostics.

### Story 3.1: Define The Knowledge-Pack Registry And Active Pack Set Contract

As a compiler engineer,  
I want Athena to define an explicit knowledge-pack registry and active pack set,  
So that component resolution is governed by deterministic package-driven inputs rather than ad hoc discovery.

**Implements:** FR-1, FR-7

**Acceptance Criteria:**

**Given** M14 introduces component-knowledge packs  
**When** the registry contract is reviewed  
**Then** active packs are defined through explicit package-governed inputs rather than filesystem order or frontend-local discovery

**Given** the active pack set is recomputed over the same repository graph  
**When** resolution runs again  
**Then** the same active knowledge-pack set is produced deterministically

### Story 3.2: Resolve Component References Through Existing Package Governance

As a compiler engineer,  
I want Athena to resolve authored component references through the existing package graph and hosted extension seams,  
So that M14 reuses M5 governance instead of inventing a parallel dependency model.

**Implements:** FR-1, FR-7

**Acceptance Criteria:**

**Given** a repository has a locked package graph  
**When** component resolution runs  
**Then** only packs available through that graph may contribute definitions

**Given** M14 is reviewed for reproducibility  
**When** package governance is inspected  
**Then** `athena.lock` remains the reproducibility anchor and no second canonical lockfile is introduced

### Story 3.3: Surface Unresolved And Conflicting Definitions Explicitly

As an architecture owner,  
I want Athena to fail explicitly on unresolved or conflicting component definitions,  
So that semantic truth is not decided silently by pack order or runtime chance.

**Implements:** FR-8

**Acceptance Criteria:**

**Given** a component reference has no matching concept or vendor implementation  
**When** resolution runs  
**Then** Athena surfaces a typed compiler-owned unresolved-definition diagnostic

**Given** two active packs define the same engineering concept or vendor part incompatibly  
**When** resolution runs  
**Then** Athena surfaces a typed compiler-owned conflict diagnostic  
**And** no implicit precedence by load order or filesystem order is allowed

## Epic 4: Downstream Integration Without New Write Authority

Runtime and platform engineers can expose resolved component knowledge to M9 and M13 consumers while preserving M8 as the only authoritative mutation path.

### Story 4.1: Publish Resolved Component Knowledge Through Runtime And Transport Seams

As a runtime engineer,  
I want resolved component knowledge to flow through existing runtime and IDE transport seams,  
So that downstream product surfaces consume Athena-owned results without opening a parallel knowledge path.

**Implements:** FR-9

**Acceptance Criteria:**

**Given** one active repository session has resolved component knowledge  
**When** the result is requested downstream  
**Then** the data is exposed through existing runtime and `ide/lsp` seams rather than by frontend-owned re-resolution

**Given** the same repository state is queried repeatedly  
**When** runtime serves the result  
**Then** the exposed payload remains deterministic and anchored to canonical semantic identities

### Story 4.2: Integrate Resolved Component Knowledge Into M9 Inputs

As a knowledge-runtime engineer,  
I want M9 layers to consume resolved component knowledge as later input,  
So that component identity, semantic ports, and physical traits can feed later derived-context or capability logic cleanly.

**Implements:** FR-4, FR-9

**Acceptance Criteria:**

**Given** M14 output is available  
**When** later M9 logic inspects the integration boundary  
**Then** resolved component knowledge is a valid input to derived-context or capability derivation  
**And** M14 still does not absorb M9 rule evaluation responsibilities

### Story 4.3: Integrate Resolved Component Knowledge Into Projection And Presentation Consumers

As a platform engineer,  
I want projection and presentation consumers to see resolved component identity and minimal physical traits,  
So that later downstream view and pack selection does not depend on thin authored labels alone.

**Implements:** FR-9

**Acceptance Criteria:**

**Given** M14 output is available  
**When** projection or presentation boundaries are reviewed  
**Then** those consumers can access resolved concept identity and minimal physical traits as downstream inputs  
**And** M14 still does not replace `Projection Model` or `Presentation IR`

### Story 4.4: Preserve M8 As The Only Mutation Authority

As an architecture owner,  
I want M14 to preserve one mutation authority across all authoring surfaces,  
So that component knowledge does not accidentally create a second write path.

**Implements:** FR-10

**Acceptance Criteria:**

**Given** M14 introduces read-only component knowledge resolution  
**When** its product and runtime boundaries are reviewed  
**Then** authored changes still converge through the M8 semantic command and mutation path before resolution runs again

**Given** future graph, form, template, AI, API, or DSL authoring is considered  
**When** product direction is inspected  
**Then** M14 explicitly preserves the one-authority write rule rather than opening a new path

## Epic 5: Proof Corpus, Product Position, And Verification Path

Product and architecture owners can prove the first electrical, Siemens-first component knowledge slice and explicitly document the multi-surface authoring position without drifting into a DSL-first or catalog-explosion product path.

### Story 5.1: Publish The First M14 Electrical Proof Corpus

As a product owner,  
I want Athena to ship one narrow electrical proof corpus for component knowledge resolution,  
So that M14 can be verified over real authored repositories instead of unit tests alone.

**Implements:** FR-8, FR-10

**Acceptance Criteria:**

**Given** the first M14 proof corpus is published  
**When** it is reviewed  
**Then** it exercises the targeted electrical proof families and Siemens-first vendor mappings

**Given** the proof corpus is used during verification  
**When** component resolution runs  
**Then** the resolved outputs remain inspectable and reproducible

### Story 5.2: Record The Product Position For Multi-Surface Authoring

As a product owner,  
I want M14 to explicitly record that DSL is canonical serialization rather than the mainstream default UI,  
So that later milestones do not drift into a DSL-first product mistake.

**Implements:** FR-10

**Acceptance Criteria:**

**Given** M14 planning and usage records are reviewed  
**When** product position is inspected  
**Then** they explicitly state that graph, forms, templates, AI, API, and DSL are producers while the semantic mutation path remains the only write authority

**Given** direct DSL authoring is considered  
**When** user roles are reviewed  
**Then** the documents position direct DSL as an expert surface rather than the required mainstream workflow

### Story 5.3: Publish The Deterministic Verification And Failure Path

As an architecture owner,  
I want Athena to publish the deterministic verification and failure path for M14,  
So that unresolved concepts, conflicting packs, and successful resolution all have explicit proof coverage.

**Implements:** FR-8

**Acceptance Criteria:**

**Given** the M14 closeout path is reviewed  
**When** verification coverage is inspected  
**Then** it includes successful concept resolution, successful vendor mapping, unresolved-definition failure, and conflicting-definition failure

**Given** the M14 proof path is repeated on the same repository state  
**When** the commands are rerun  
**Then** the same outputs and diagnostics are produced deterministically
