# Athena - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for Athena, decomposing the requirements from the PRD, UX Design if it exists, and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

FR-1: Athena can host plugin-declared domain schema for entities, properties, ports, and related domain capabilities.
FR-2: Athena can register plugin contributions for domain semantics, validation, compiler participation, and renderer contribution through stable contracts.
FR-3: Athena can discover, initialize, inspect, and shut down hosted plugins in a governed way.
FR-4: Athena keeps parsing and core semantic ownership generic while allowing plugins to provide domain meaning through declared schema and passes.
FR-5: Athena exposes compilation as an explicit pass pipeline rather than one opaque compiler blob.
FR-6: Athena can let a domain plugin contribute domain semantics to lowering or later semantic passes without ceding kernel ownership of canonical model rules.
FR-7: Athena continues to own generic validation such as duplicate identifiers, missing references, and invalid generic graph structure.
FR-8: Athena can host plugin-owned validation rules for domain-specific constraints.
FR-9: Athena keeps renderer contracts and backend orchestration generic.
FR-10: Athena can host plugin-contributed rendering behavior for domain-specific symbols or visual consequences.
FR-11: Athena refactors the existing `domain-electrical` extension so it proves the stable M3 SPI rather than bypassing it.
FR-12: Athena provides a minimal `domain-dummy` plugin to prove the SPI is not electrical-specific.
FR-13: Athena can verify kernel behavior across different hosted plugin sets.
FR-14: Athena can add a new proof domain after the SPI freeze without requiring further kernel domain edits.

### NonFunctional Requirements

NFR-1: Kernel modules remain generic and domain-agnostic in responsibility.
NFR-2: The same authored inputs, hosted plugin set, and pass ordering produce the same outcomes.
NFR-3: Hosted plugin inventory, contribution points, and pass participation are visible enough to debug and govern.
NFR-4: M3 proves extensibility on the Java 25 JVM-first path before broader dynamic or cross-platform plugin delivery.
NFR-5: M3 must not architecturally block future local-directory or remote-URL plugin loading, even though that is out of scope for this milestone.

### Additional Requirements

- M3 is an extensibility milestone, not a Studio milestone.
- The current ServiceLoader host remains the first plugin-source proof, but plugin hosting must be split into source and approval layers so future loading mechanisms can be added without bypassing governance.
- The authored DSL remains the source of truth and stays structurally generic in M3; plugins extend interpretation, not grammar.
- `Engineering IR` remains the canonical semantic model inherited from M0 and reaffirmed by M1 and M2.
- Runtime ownership established in M1 remains intact: lifecycle, hosted service orchestration, and semantic mutation through commands stay runtime-owned.
- Layout and geometry boundaries proven in M2 remain downstream consequences and may not become plugin-owned semantic authority.
- Stable plugin contracts must move into a dedicated kernel API boundary instead of remaining hidden inside compiler implementation packages.
- The compiler must be visibly governed as a named pass pipeline with deterministic approved-plugin ordering.
- Kernel validation and domain validation must remain separate, with generic rules preserved even when no domain plugin is present.
- Kernel renderer orchestration remains generic and downstream; domain plugins contribute presentation behavior only through approved contracts.
- The existing `:extensions:domain-electrical` module is the first real proof domain and must be refactored into the stable M3 SPI.
- The electrical proof vocabulary remains intentionally narrow: `Motor`, `Lamp`, `Switch`, `Wire`.
- A synthetic `:extensions:domain-dummy` plugin is required to prove the SPI is not secretly electrical-shaped.
- M3 must publish an automated verification matrix for zero plugins, electrical only, dummy only, and both together.
- Java 25 remains mandatory and Gradle verification on Windows remains sequential.
- Dynamic local-directory loading, remote URL acquisition, hot load/unload, plugin marketplaces, and plugin-defined grammar extensions remain explicitly deferred.

### UX Design Requirements

None identified. UX work is explicitly out of scope for M3 except where existing desktop or CLI surfaces are needed to inspect hosted extensibility behavior.

### FR Coverage Map

FR-1: Epic 1 - define the stable hosted extension surface for plugin-declared domain schema
FR-2: Epic 1 - register typed plugin contributions through stable contracts and governed approval
FR-3: Epic 1 - prove hosted plugin lifecycle and inspectable inventories on the JVM-first path
FR-4: Epic 1 - keep the authored DSL and kernel core generic while domains extend interpretation
FR-5: Epic 1 - refactor compilation into an explicit pass pipeline
FR-6: Epic 1 - support domain semantic contributions through governed compiler stages
FR-7: Epic 2 - preserve kernel-owned generic validation while domains supply their own rules
FR-8: Epic 2 - support plugin-owned domain validation
FR-9: Epic 2 - keep renderer contracts and orchestration generic
FR-10: Epic 2 - support plugin-contributed rendering behavior
FR-11: Epic 2 - refactor the existing electrical extension into the M3 proof shape
FR-12: Epic 2 - add a second synthetic dummy proof plugin
FR-13: Epic 3 - automate the zero/one/multi-plugin verification matrix
FR-14: Epic 3 - prove that new proof domains can be introduced after the SPI freeze without fresh kernel domain edits

## Epic List

### Epic 1: Prove The Hosted Extensibility Platform
Athena provides a stable hosted extension surface so extension authors can add domain meaning through approved contracts, governed plugin hosting, and an explicit pass pipeline without turning the kernel domain-specific.
**FRs covered:** FR-1, FR-2, FR-3, FR-4, FR-5, FR-6

### Epic 2: Prove External Domains Can Live Outside The Kernel
Athena proves that real domain behavior can stay outside kernel code by externalizing domain validation and rendering, refactoring the existing electrical extension into the stable SPI, and adding a second synthetic proof domain.
**FRs covered:** FR-7, FR-8, FR-9, FR-10, FR-11, FR-12

### Epic 3: Prove The Kernel Boundary Holds Under Verification
Athena demonstrates the stable-kernel claim with an automated verification matrix across zero-plugin, single-plugin, and multi-plugin hosted states and makes regressions back toward kernel-owned domain logic visible.
**FRs covered:** FR-13, FR-14

## Epic 1: Prove The Hosted Extensibility Platform

Athena provides a stable hosted extension surface so extension authors can add domain meaning through approved contracts, governed plugin hosting, and an explicit pass pipeline without turning the kernel domain-specific.

As a platform engineer,
I want stable plugin-facing contracts moved into a dedicated kernel API boundary,
So that extensions can target an official SPI instead of depending on compiler internals.

**Acceptance Criteria:**

**Given** the current workspace still hosts plugin contracts inside compiler implementation areas
**When** the M3 SPI boundary is introduced
**Then** Athena adds a dedicated kernel plugin API module or equivalent stable kernel-owned boundary for plugin-facing contracts
**And** extensions no longer need to depend on compiler implementation packages to exist

**Given** the new API boundary is created
**When** stable plugin-facing contracts are moved
**Then** manifest metadata, extension-point declarations, and other durable hosted plugin contracts live in that boundary
**And** contracts that are compiler-private or runtime-private do not remain mixed into the stable SPI surface

**Given** the SPI boundary must preserve earlier milestones
**When** module dependencies are reviewed
**Then** the new boundary keeps `Engineering IR`, runtime ownership, and projection ownership inside kernel-owned modules
**And** M0, M1, and M2 proof modules continue to build without regressing their semantic authority rules

**Given** the new API boundary is implemented
**When** the standard Java `25` build and module checks are executed
**Then** the workspace builds successfully with the dedicated plugin API boundary in place
**And** shared build versions continue to be managed through `gradle/libs.versions.toml`
