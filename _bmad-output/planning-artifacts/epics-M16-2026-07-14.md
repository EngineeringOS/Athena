# Athena - Epic Breakdown

## Overview

This document provides the epic and story breakdown for Athena M16, decomposing semantic macro and governed reuse into implementable work above the completed M15 guided-authoring foundation.

M16 is a semantic reuse milestone, not a package-system expansion and not a graphic-macro milestone.

Its job is to prove that Athena can deliver governed engineering assembly reuse through:

- Semantic Macro contracts
- reusable component and connection templates
- repository-governed macro loading and catalog publishing
- parameterized instantiation and deterministic preview
- accepted expansion through M8
- origin traceability and membership inspection

while preserving:

- M5 as the only repository and package governance layer
- M8 as the only mutation authority
- M14 as the source of component and port knowledge
- M13 as the downstream presentation layer rather than macro truth

## Requirements Inventory

### Functional Requirements

FR-1: Athena can define Semantic Macro as a governed reusable assembly contract.
FR-2: Athena can publish reuse contracts as dedicated platform models.
FR-3: Athena can validate macro parameters before expansion.
FR-4: Athena can build a deterministic expansion preview.
FR-5: Athena can route expansion acceptance through the sole mutation authority.
FR-6: Athena can persist accepted expansion into canonical engineering state.
FR-7: Athena can expose a governed reuse catalog from active repository context.
FR-8: Athena can support parameter editing and review-first acceptance in the workbench.
FR-9: Athena can preserve macro origin on expanded semantic subjects.
FR-10: Athena can expose expansion membership and origin inspection.
FR-11: Athena can reuse existing M5 governance instead of inventing a new package system.
FR-12: Athena can prove one narrow electrical reuse slice end to end.

### NonFunctional Requirements

NFR-1: M16 introduces no second mutation path outside M8.
NFR-2: M16 introduces no second package resolver or second lockfile outside M5.
NFR-3: Semantic Macro truth remains semantic-first and may not collapse into graphics truth.
NFR-4: Preview and acceptance remain deterministic and inspectable.
NFR-5: Origin traceability remains machine-readable across source, graph, inspection, and review surfaces.
NFR-6: Workbench surfaces remain consumers of platform-owned reuse services.
NFR-7: Presentation hints remain downstream and replaceable.
NFR-8: The first proof stays narrow, electrical, and repository-scoped.

### Additional Requirements

- M16 builds on completed M5 repository and package governance and must not redefine package ownership.
- M16 builds on completed M8 mutation authority and may not bypass it from catalog, parameter, or preview flows.
- M16 builds on completed M14 component knowledge and must consume concept identity, semantic ports, minimal physical traits, and vendor implementations from that layer.
- M16 builds on completed M15 guided authoring and should keep reuse surfaces as thin consumers of shared runtime services.
- `Presentation IR` and renderer systems remain downstream consumers and may not become macro truth.
- The first proof should stay narrow around `DOL Starter`, `PLC Rack`, and `24V Distribution Unit`.
- The first proof should preserve explicit origin identity, parameter values, and expansion membership for all expanded semantic subjects.
- M16 excludes marketplace federation, arbitrary graphic-block reuse, final schematic generation, and full update or rebind lifecycle.

### UX Design Requirements

- The first workbench proof should expose a governed reuse catalog or assembly palette.
- The first workbench proof should expose parameter editing for one selected Semantic Macro.
- The workbench should show deterministic preview consequences before acceptance.
- The workbench should allow origin and membership inspection after acceptance.
- The proof flow should not require raw DSL editing for the happy path.

### FR Coverage Map

FR-1: Epic 1 and Epic 2 - reuse contracts and governed catalog definitions
FR-2: Epic 1 - dedicated platform model boundaries
FR-3: Epic 3 - parameter schema and validation
FR-4: Epic 3 - deterministic preview
FR-5: Epic 4 - M8-backed acceptance
FR-6: Epic 4 - canonical persistence and downstream refresh
FR-7: Epic 2 and Epic 5 - governed catalog and first proof slice
FR-8: Epic 3 and Epic 5 - parameter editing and review-first workbench proof
FR-9: Epic 4 - origin traceability
FR-10: Epic 4 and Epic 5 - membership and origin inspection
FR-11: Epic 2 - reuse of M5 governance
FR-12: Epic 5 - narrow electrical proof slice and verification

## Epic 1: Semantic Macro And Template Contract Foundation

Platform engineers can define Semantic Macro and template contracts as dedicated platform models so assembly reuse stops being expressed as package hacks, graphic blocks, or frontend-local templates.

### Story 1.1: Publish The Core Semantic Macro Contract

As a platform engineer,
I want Athena to define a Semantic Macro as a first-class reusable assembly contract,
So that governed engineering reuse has a stable identity above package metadata and below runtime expansion.

**Implements:** FR-1, FR-2

**Acceptance Criteria:**

**Given** M16 introduces semantic reuse
**When** the contract is reviewed
**Then** Athena defines Semantic Macro identity, instantiation identity, parameter schema, preview, accepted-expansion, and origin contracts in dedicated platform models

**Given** Semantic Macro ownership is inspected
**When** architecture owners compare it with package and UI layers
**Then** the contract remains distinct from M5 package contracts and from frontend-local widget models

### Story 1.2: Define Reusable Template Payload Contracts

As a platform engineer,
I want Athena to define reusable template payloads for components and connections,
So that macro expansion is expressed as semantic template composition rather than as graphics or opaque generators.

**Implements:** FR-1, FR-2

**Acceptance Criteria:**

**Given** template payload contracts are reviewed
**When** the boundary is inspected
**Then** Athena defines reusable component-template and connection-template contracts in a dedicated template model

**Given** the template layer is compared with presentation systems
**When** reviewers inspect ownership
**Then** templates may include optional downstream hints
**And** they do not make SVG, manual layout, or symbol geometry the engineering source of truth

### Story 1.3: Publish Shared Reuse Runtime And Transport Seams

As a runtime engineer,
I want Semantic Macro operations to flow through shared runtime and transport seams,
So that reuse services remain platform-owned and reusable by catalog, forms, AI, or later APIs.

**Implements:** FR-2

**Acceptance Criteria:**

**Given** the first M16 implementation boundaries are reviewed
**When** runtime ownership is inspected
**Then** catalog lookup, parameter validation, preview generation, acceptance, and origin inspection are exposed through Athena-owned runtime and `ide/lsp` seams

**Given** future reuse entrypoints are considered
**When** the service boundary is checked
**Then** the contract does not depend on one specific Theia panel or graph widget

## Epic 2: Governed Macro Loading And Catalog

Runtime and platform engineers can resolve active Semantic Macros through the existing repository and package graph and publish them through a governed catalog so reuse availability is reproducible and inspectable.

### Story 2.1: Resolve Active Semantic Macros Through Existing Package Governance

As a compiler engineer,
I want Athena to resolve active Semantic Macros only from the governed repository and package graph,
So that M16 reuses M5 authority instead of inventing a second package system.

**Implements:** FR-7, FR-11

**Acceptance Criteria:**

**Given** a repository has a locked package graph
**When** active Semantic Macros are resolved
**Then** only packs available through that graph may contribute macro definitions

**Given** M16 reproducibility is reviewed
**When** package-governance boundaries are inspected
**Then** `athena.lock` remains the reproducibility anchor
**And** no second lockfile or ad hoc macro resolver is introduced

### Story 2.2: Publish The Runtime-Owned Macro Catalog

As a runtime engineer,
I want Athena to publish one active Semantic Macro catalog from repository context,
So that the workbench consumes governed availability instead of hardcoded menus.

**Implements:** FR-1, FR-7, FR-11

**Acceptance Criteria:**

**Given** one governed repository session is active
**When** the macro catalog is requested
**Then** Athena returns available Semantic Macros from runtime-owned repository context rather than frontend-local discovery

**Given** the same repository state is reopened
**When** catalog generation runs again
**Then** the same catalog entries are produced deterministically

### Story 2.3: Deliver The First Governed Reuse Catalog Surface

As an engineer,
I want Athena to show available Semantic Macros in a dedicated reuse catalog,
So that I can start assembly reuse without authoring raw DSL or navigating package internals.

**Implements:** FR-7

**Acceptance Criteria:**

**Given** the first M16 workbench proof is implemented
**When** the reuse catalog is displayed
**Then** it lists available Semantic Macros from the runtime-owned catalog rather than from hardcoded frontend lists

**Given** the first proof slice is electrical only
**When** catalog entries are reviewed
**Then** the surface can show at least `DOL Starter`, `PLC Rack`, and `24V Distribution Unit`

## Epic 3: Parameterized Instantiation And Deterministic Preview

Engineers can configure one Semantic Macro and review deterministic expansion consequences before acceptance so reuse remains inspectable instead of opaque generation.

### Story 3.1: Define Parameter Schema And Validation For Macro Instantiation

As a platform engineer,
I want Athena to define explicit parameter schema and validation behavior for Semantic Macros,
So that engineers configure meaningful engineering values instead of editing generated output after the fact.

**Implements:** FR-3

**Acceptance Criteria:**

**Given** one Semantic Macro declares parameters
**When** its schema is reviewed
**Then** Athena can express required values, defaults, validation rules, and stable parameter ids

**Given** a user provides invalid or incomplete values
**When** instantiation is attempted
**Then** Athena blocks preview generation until validation succeeds

### Story 3.2: Build Deterministic Expansion Preview For Configured Instantiations

As a runtime engineer,
I want Athena to generate a deterministic preview of one configured Semantic Macro,
So that engineers can inspect semantic consequences before acceptance.

**Implements:** FR-4

**Acceptance Criteria:**

**Given** one Semantic Macro and one valid parameter set
**When** preview generation runs
**Then** Athena returns the components, ports, connections, origin anchors, and presentation consequences that would result from acceptance

**Given** the same repository state and parameter set
**When** preview generation runs again
**Then** the same preview is produced deterministically

### Story 3.3: Support Parameter Editing And Review-First Acceptance In The Workbench

As an engineer,
I want to edit Semantic Macro parameters and review the preview in the workbench,
So that reuse feels like one governed product flow instead of a hidden generator.

**Implements:** FR-8

**Acceptance Criteria:**

**Given** one catalog entry is selected
**When** the engineer edits meaningful parameters
**Then** the workbench submits those values through shared reuse services rather than local expansion logic

**Given** preview is displayed
**When** the engineer approves or cancels
**Then** the flow remains review-first
**And** cancel leaves canonical state unchanged

## Epic 4: Accepted Expansion And Traceable Canonical State

Runtime and platform engineers can commit an approved Semantic Macro preview through M8 and preserve reusable origin facts so accepted reuse becomes canonical engineering state rather than hidden generated output.

### Story 4.1: Convert Approved Preview Into One Governed Mutation Bundle

As a runtime engineer,
I want Athena to convert one approved Semantic Macro preview into one governed mutation bundle,
So that reuse acceptance still flows through M8 as the only write authority.

**Implements:** FR-5

**Acceptance Criteria:**

**Given** a preview is approved
**When** acceptance is handed off
**Then** Athena emits one M8-facing mutation bundle rather than direct source or graph writes

**Given** the bundle contains multiple low-level mutations
**When** acceptance semantics are reviewed
**Then** the operation remains atomic at the user-intent level

### Story 4.2: Persist Accepted Expansion Into Canonical State And Downstream Refresh

As a product owner,
I want accepted Semantic Macro expansion to refresh canonical and downstream state coherently,
So that reused assemblies behave like first-class engineered structure.

**Implements:** FR-6

**Acceptance Criteria:**

**Given** one approved expansion has been committed through M8
**When** canonical rebuild completes
**Then** source, graph, semantic inspection, and review state all refresh coherently

**Given** accepted output is inspected
**When** its representation is reviewed
**Then** the result remains machine-readable semantic structure rather than an opaque generated blob

### Story 4.3: Preserve Origin Traceability And Expansion Membership

As an architecture owner,
I want Athena to preserve origin traceability and expansion membership for accepted reuse,
So that later diff, replace, review, and AI reasoning work can stay grounded.

**Implements:** FR-9, FR-10

**Acceptance Criteria:**

**Given** one Semantic Macro expansion has been accepted
**When** origin data is inspected
**Then** Athena records Semantic Macro identity, instantiation identity, and parameter values for that expansion

**Given** expanded semantic subjects are inspected
**When** membership is queried
**Then** Athena can identify which semantic subjects belong to the accepted expansion

## Epic 5: Narrow Electrical Proof Slice And Verification Path

Product and architecture owners can prove one repository-scoped electrical reuse slice end to end so M16 closes with evidence instead of only contracts and UI shells.

### Story 5.1: Publish The First Electrical Semantic Macro Proof Slice

As an electrical platform owner,
I want Athena to ship the first narrow electrical Semantic Macro slice,
So that M16 proves reusable engineering assemblies over real engineering examples.

**Implements:** FR-7, FR-12

**Acceptance Criteria:**

**Given** the first M16 proof slice is published
**When** reusable assemblies are reviewed
**Then** the slice includes at least `DOL Starter`, `PLC Rack`, and `24V Distribution Unit`

**Given** the proof slice is compared with M14 component knowledge
**When** expansion inputs are inspected
**Then** the reusable assemblies derive from governed component, connection, and vendor-implementation knowledge rather than handwritten graphics truth

### Story 5.2: Expose Origin And Membership Inspection Across Product Surfaces

As an engineer,
I want Athena to reveal macro origin and expansion membership across product surfaces,
So that accepted reuse stays inspectable after the initial preview is gone.

**Implements:** FR-8, FR-10

**Acceptance Criteria:**

**Given** one accepted expansion exists
**When** the engineer inspects source, graph, or review surfaces
**Then** Athena can reveal which Semantic Macro created the structure and which parameter values were used

**Given** the engineer selects one expanded subject
**When** origin or membership inspection is requested
**Then** Athena can identify the accepted expansion that subject belongs to

### Story 5.3: Publish The Deterministic Verification Path For M16

As a platform owner,
I want Athena to publish one deterministic verification path for semantic reuse,
So that M16 closes with repository-backed proof instead of narrative claims.

**Implements:** FR-12

**Acceptance Criteria:**

**Given** the M16 proof path is reviewed
**When** the scenario is inspected
**Then** it covers open repository -> choose `DOL Starter` -> configure parameters -> preview expansion -> accept -> verify source, graph, inspection, review, and origin coherence

**Given** the same proof repository and parameter set are reused
**When** the verification path is rerun
**Then** the same preview, accepted structure, and traceability outputs are produced deterministically
