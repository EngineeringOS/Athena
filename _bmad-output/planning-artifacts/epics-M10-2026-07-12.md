# Athena - Epic Breakdown

## Overview

This document provides the epic and story breakdown for Athena M10, decomposing AI-assisted engineering reasoning into governed, implementable work above the completed M9 knowledge runtime.

## Requirements Inventory

### Functional Requirements

FR-1: Athena can build one deterministic AI reasoning context package from governed semantic outputs.
FR-2: Athena can preserve traceable audit data and cited evidence for each reasoning proposal.
FR-3: Athena can generate grounded explanations for governed engineering diagnostics.
FR-4: Athena can summarize engineering impact for semantic review and SCM flows.
FR-5: Athena can suggest narrow review-ready next checks or remediation direction without auto-applying change.
FR-6: Athena can keep AI outputs assistive and non-authoritative.
FR-7: Athena can preserve explicit accepted, dismissed, unresolved, and unavailable proposal states.
FR-8: Athena can deliver reasoning through existing Athena surfaces and additive Theia AI integration.
FR-9: Athena can keep the AI provider boundary replaceable and downstream of Athena-owned reasoning contracts.

### NonFunctional Requirements

NFR-1: Canonical engineering truth remains upstream of any provider call.
NFR-2: The same request kind over the same semantic state yields the same reasoning context package.
NFR-3: Proposal audit data remains inspectable and tied to governed evidence.
NFR-4: Provider transport remains replaceable without changing semantic contracts.
NFR-5: M10 stays narrow and does not turn into autonomous editing or broad chat sprawl.
NFR-6: Existing Athena semantic surfaces remain the first delivery path.
NFR-7: Theia AI packages are reused for generic IDE AI concerns, not engineering truth ownership.

### Additional Requirements

- M10 builds directly on completed M9 derived context, capability facts, constraint evaluation, diagnostics, and impact consequences.
- Existing M6 review and commit semantics remain the review substrate for M10 summaries and suggestions.
- Existing M4 through M8 Athena Theia product shell remains the desktop host for M10.
- Current `AthenaAiProposalRuntimeService` is only groundwork; M10 must generalize from command-only proposals to governed reasoning proposals.
- Runtime owns reasoning-context assembly, proposal ids, decision state, and audit snapshots.
- `ide/lsp` remains the sole IDE semantic and reasoning transport path.
- Theia AI package adoption must stay additive and product-shell scoped.
- First proof flows are:
  - diagnostic explanation
  - engineering impact summary
  - review-ready next-check suggestion
- Mock-provider verification is mandatory before trusting hosted or local real providers.
- M10 must publish proof corpus and operator docs that clearly state what AI can and cannot do.

### FR Coverage Map

FR-1: Epic 1 - governed reasoning contracts and deterministic context assembly
FR-2: Epic 1 and Epic 2 - typed evidence, audit, and proposal traceability
FR-3: Epic 2 - grounded diagnostic explanation
FR-4: Epic 2 - engineering impact summary
FR-5: Epic 2 - review-ready next checks
FR-6: Epic 1 and Epic 3 - non-authoritative proposal lifecycle and UI
FR-7: Epic 3 - explicit proposal decision state through current workbench seams
FR-8: Epic 3 - Theia AI reuse and additive product integration
FR-9: Epic 1 and Epic 4 - provider boundary and proof-path verification

## Epic 1: Governed AI Reasoning Contracts

Platform engineers can define typed reasoning requests, deterministic context assembly, and provider-neutral proposal lifecycle above M9 outputs without giving AI semantic authority.

### Story 1.1: Publish Governed AI Reasoning Contracts

As a platform engineer,
I want Athena to publish typed reasoning request, evidence, proposal, and decision-state contracts,
so that AI assistance can sit above governed semantic outputs without hiding inside provider-specific payloads.

**Implements:** FR-1, FR-2, FR-6, FR-7, FR-9

**Acceptance Criteria:**

1. Given M9 knowledge-runtime outputs already exist, when M10 defines its first AI boundary, then Athena publishes typed reasoning nouns for request category, reasoning context, cited evidence, proposal category, provider result status, and proposal decision state, and those contracts remain Athena-owned rather than vendor-owned.
2. Given one reasoning proposal is inspected later, when Athena reads the recorded contract state, then it can identify what governed evidence was used, what response category was produced, and whether the proposal is accepted, dismissed, unresolved, or unavailable.

### Story 1.2: Derive Deterministic Reasoning Contexts From Governed Semantic Outputs

As a platform engineer,
I want Athena to assemble deterministic reasoning context packages from M9 and M6 outputs,
so that the same engineering state always yields the same AI input package for the same request kind.

**Implements:** FR-1, FR-2, FR-6

**Acceptance Criteria:**

1. Given the same semantic state and request kind, when Athena assembles a reasoning context package twice, then both packages are structurally identical and preserve deterministic ordering.
2. Given a reasoning request for diagnostic explanation, impact summary, or next-check suggestion, when Athena assembles the context, then it uses canonical subject identities, derived context, capability facts, diagnostics, impact consequences, and review facts where available instead of raw frontend-only state.

### Story 1.3: Host Provider-Neutral Reasoning Sessions And Typed Outcome States

As a platform engineer,
I want Athena runtime to own reasoning session orchestration and typed provider outcomes,
so that hosted, local, or mock providers can plug in without redefining proposal lifecycle rules.

**Implements:** FR-2, FR-6, FR-7, FR-9

**Acceptance Criteria:**

1. Given a reasoning request is submitted, when Athena routes it through the provider boundary, then runtime records a provider-neutral session result such as success, unavailable, or failed without changing canonical engineering truth.
2. Given provider integration changes later, when M10 contracts are inspected, then provider transport remains downstream of Athena-owned reasoning session and proposal models.

### Story 1.4: Expose Reasoning Requests Through Athena LSP As Sole IDE Boundary

As a platform engineer,
I want Athena to expose governed reasoning requests and results through `ide/lsp`,
so that IDE surfaces can consume reasoning proposals without importing JVM kernel code directly.

**Implements:** FR-1, FR-2, FR-8, FR-9

**Acceptance Criteria:**

1. Given one reasoning request is initiated from the workbench, when Athena transports it to the backend, then the request and result flow only through Athena LSP and runtime-owned contracts.
2. Given reasoning transport payloads are reviewed, when architecture boundaries are inspected, then frontend payloads remain transport DTOs rather than the source of proposal truth or evidence assembly.

## Epic 2: First Explanation And Review Proofs

Engineers and reviewers can ask Athena for grounded diagnostic explanations, impact summaries, and narrow review-ready next checks built from governed evidence.

### Story 2.1: Generate Grounded Diagnostic Explanation Proposals

As an engineer,
I want Athena to explain why a governed engineering diagnostic happened,
so that I can understand the relevant facts without interpreting raw semantic output alone.

**Implements:** FR-2, FR-3, FR-6

**Acceptance Criteria:**

1. Given a governed engineering sufficiency diagnostic exists, when Athena requests an explanation, then the returned proposal references the relevant derived context, capability fact, or constraint result rather than generic advice only.
2. Given the explanation is reviewed, when the cited evidence is inspected, then Athena can show which governed facts informed the explanation.

### Story 2.2: Generate Engineering Impact Summary Proposals

As a reviewer,
I want Athena to summarize downstream engineering impact for a governed change,
so that I can see what to inspect next without manually reconstructing consequence chains.

**Implements:** FR-2, FR-4, FR-6

**Acceptance Criteria:**

1. Given a semantic review or SCM context already exists, when Athena requests an impact summary, then the summary distinguishes direct edits from downstream affected subjects.
2. Given the summary is inspected later, when audit data is read, then the cited impact facts and review facts remain reconstructable.

### Story 2.3: Generate Review-Ready Next-Check Proposals

As a reviewer,
I want Athena to suggest narrow next checks or remediation direction,
so that I can review governed engineering change faster without auto-applying anything.

**Implements:** FR-5, FR-6

**Acceptance Criteria:**

1. Given the current proof slice surfaces a protection, cable, or relay review concern, when Athena asks for next checks, then the returned proposal stays advisory and review-ready rather than executable mutation.
2. Given no safe next check can be produced, when Athena records the outcome, then proposal state becomes unresolved or unavailable instead of inventing a weak suggestion.

### Story 2.4: Preserve Cited-Fact Audit Trails Across Reasoning Proofs

As an architecture owner,
I want every explanation, summary, and next-check proof to preserve cited evidence and audit metadata,
so that AI assistance remains inspectable and governed.

**Implements:** FR-2, FR-3, FR-4, FR-5

**Acceptance Criteria:**

1. Given any M10 reasoning proposal exists, when it is inspected, then Athena can show request category, proposal category, cited evidence, provider outcome, and decision state.
2. Given a provider call fails or is unavailable, when Athena records the result, then the audit trail preserves the failure state without fabricating semantic content.

## Epic 3: Additive Product Surface Integration

Athena can surface reasoning actions and proposal decisions through current semantic panels and Theia AI foundation without redesigning the whole product shell.

### Story 3.1: Integrate Theia AI Foundation Into Athena Product Shell

As a platform engineer,
I want Athena product packages to reuse Theia AI dependencies and registration seams,
so that provider configuration and generic assistant chrome come from a standard foundation.

**Implements:** FR-8, FR-9

**Acceptance Criteria:**

1. Given Athena desktop product packaging is reviewed, when M10 foundation is added, then Theia AI packages are integrated additively into product and frontend dependencies without bypassing Athena LSP.
2. Given provider configuration is required, when operators inspect the product shell, then configuration and provider management use Theia-owned generic capabilities rather than a custom Athena-only reinvention.

### Story 3.2: Surface Reasoning Actions In Semantic Inspection And Review Panels

As an engineer,
I want Athena to expose reasoning actions beside current diagnostics and review facts,
so that I can request explanation or summary where the governed evidence already lives.

**Implements:** FR-3, FR-4, FR-8

**Acceptance Criteria:**

1. Given semantic inspection or semantic review data is already visible, when I invoke an M10 reasoning action, then Athena requests the matching reasoning flow without opening a separate semantic subsystem.
2. Given the action returns a result, when the panel updates, then the returned proposal remains anchored to the currently inspected diagnostic, subject, or review context.

### Story 3.3: Preserve Explicit Proposal Decision State In Existing Athena Workbench Seams

As a reviewer,
I want proposal decisions to stay explicit inside the existing workbench,
so that accepted, dismissed, unresolved, and unavailable outcomes are visible and inspectable.

**Implements:** FR-6, FR-7, FR-8

**Acceptance Criteria:**

1. Given one reasoning proposal is visible in Athena, when the operator accepts or dismisses it, then the decision state is recorded and visible through the same product seams.
2. Given a proposal is unresolved or unavailable, when the user revisits it, then Athena preserves that state without pretending a successful recommendation exists.

## Epic 4: Proof Corpus And Safety Boundaries

Architecture owners can validate M10 with deterministic proof fixtures, mock-provider verification, and explicit operator boundary documentation.

### Story 4.1: Publish M10 Proof Corpus And Deterministic Mock-Provider Verification Path

As an architecture owner,
I want Athena to ship repeatable M10 proof fixtures with a mock provider,
so that governance and contract behavior can be verified deterministically before relying on live models.

**Implements:** FR-1, FR-2, FR-9

**Acceptance Criteria:**

1. Given M10 core flows exist, when the proof corpus is executed, then Athena can verify diagnostic explanation, impact summary, and next-check flows through a deterministic mock-provider path.
2. Given verification is rerun later, when the same proof fixtures are used, then the context packages and expected proposal states remain stable.

### Story 4.2: Document Provider Configuration, Safety Boundaries, And Operator Usage

As a product owner,
I want M10 to publish clear usage and safety docs,
so that operators know exactly what AI can do, cannot do, and how to configure supported providers.

**Implements:** FR-6, FR-8, FR-9

**Acceptance Criteria:**

1. Given M10 is reviewed or demonstrated, when operators read the docs, then they can see supported flows, provider setup, mock-provider verification, and non-goals clearly.
2. Given the AI layer is audited, when the safety notes are inspected, then the boundary against autonomous mutation authority is explicit and testable.
