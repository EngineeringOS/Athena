topic: Athena M3 Kernel Extensibility Proof
updated: 2026-07-07T10:24
---

- (decision) M3 may refactor kernel contracts once to establish a stable SPI and pass-pipeline boundary; after that, adding a new domain should be extension work.
- (decision) M3 is defined as the kernel extensibility proof after M0 to M2, not as a UI or broader runtime milestone.
- (decision) The kernel should stay small by keeping parsing and core ownership generic while plugins supply domain schema, validation, rendering, and governed pass contributions.
- (decision) The first plugin-loading proof remains the current JVM-first ServiceLoader host, while future dynamic local-directory and remote-URL loading stays out of M3 delivery scope.
- (decision) The existing domain-electrical extension is the first real proof domain and must be refactored into the stable M3 SPI rather than replaced by a parallel implementation.
- (decision) M3 architecture freezes a small-kernel boundary: generic language, canonical semantics, runtime, projection, and backend orchestration remain kernel-owned while domain meaning lives in plugins.
- (decision) The authored DSL remains generic in M3; plugin-hosted domains extend interpretation through schema, validation, render contribution, and governed pass participation rather than through new grammar.
- (decision) Stable plugin contracts move into a dedicated kernel API boundary and hosted discovery is split into plugin source and approval layers so ServiceLoader remains the first source without becoming the permanent architecture.
- (decision) The compiler is governed as a named pass pipeline with plugin participation only at declared stage contribution points in deterministic approved-plugin order.
- (decision) The M3 proof requires both the refactored domain-electrical extension and a synthetic domain-dummy extension plus an automated zero/one/multi-plugin verification matrix.
- (event) architecture spine drafted
