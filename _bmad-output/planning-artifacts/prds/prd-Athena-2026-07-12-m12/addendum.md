# M12 Addendum

This addendum captures useful M12 planning detail that informs architecture and epic shaping but is intentionally more implementation-shaped than the main PRD body.

## 1. Agreed Milestone Position

The active planning sequence is now:

- **M11** - first serious electrical ECAD workbench depth
- **M12** - electrical renderer correctness and operator-density hardening

This means M12 is not:

- another knowledge-runtime milestone
- another standards or policy milestone
- a broad AI milestone
- a full EPLAN clone
- a freeform drawing milestone

It is the milestone where Athena must prove that the first electrical product surface can render and behave credibly under operator scrutiny.

## 2. Why The Current Foundation Is Already Strong Enough

Athena already has:

- canonical electrical semantics
- runtime-owned projection sessions
- graph-first workbench delivery
- governed source and graph mutation
- electrical knowledge diagnostics and impact
- AI-assisted reasoning surfaces
- richer electrical projection families, sheets, notation packs, and repeated references

So the current question is no longer:

- can Athena represent electrical meaning?

The current question is:

- can Athena render and present that meaning in a way electrical users can trust?

## 3. What Is Still Missing Today

The current workspace does not yet prove:

- connection rendering that clearly reads as conductor intent
- robust endpoint anchoring under denser scenes
- reliable fit-to-viewport and scene focus on larger proofs
- enough routing and overlap quality to avoid generic graph appearance
- cross-reference and related-subject movement that feels native to electrical operator work
- dense and disciplined Athena-owned support panels around the graph

That is why this milestone exists.

## 4. Product Reference Rule

QElectroTech and EPLAN should be treated as:

- renderer readability references
- workflow-density references
- operator-panel references
- electrical notation references

They should not be treated as:

- semantic truth models
- architecture centers
- reasons to move authority into the renderer

The right attitude is:

- learn how serious electrical tools make dense work readable
- keep Athena's semantic-layer strategy intact

## 5. Recommended First M12 Proof Shape

The first renderer-correctness proof should likely include:

- one denser electrical repository carried forward from M11
- at least one view where conductor readability is visibly testable
- endpoint-heavy cases where anchoring errors would be obvious
- one first cross-reference navigation flow such as coil to contact or source to destination
- panel-density checks around hierarchy, diagnostics, and inspection
- explicit validation steps for fit, zoom, focus, and scene recovery
- at least one larger readability benchmark tier beyond the baseline dense case

That is large enough to expose real renderer weaknesses without exploding into a full parity program.

## 6. Renderer Boundary

The renderer should stay downstream of:

- canonical engineering identity
- electrical projection-family contracts
- sheet and notation contracts
- runtime-owned projection sessions

That means:

- coordinates are not engineering truth
- routing heuristics are not engineering truth
- visual grouping is not engineering truth
- the renderer may improve readability, but it may not invent semantics

The routing split should stay explicit:

```text
Semantic Layer:
    endpoint ownership

Projection Layer:
    preferred routing corridor

Renderer:
    actual visual path
```

That keeps route geometry out of the truth model while still allowing visual improvement.

## 7. Workbench Quality Targets

M12 should improve:

- electrical connection readability
- fit-to-view reliability
- visible endpoint clarity
- cross-reference and related-reveal navigation trust
- control compactness around the graph
- dense panel presentation in Athena-owned surfaces

But it should avoid:

- decorative UI churn
- broad design-system work
- introducing a second frontend-owned graph model
- coupling the kernel to one visual framework decision too early

## 8. Risks That Must Stay Explicit

- renderer heuristics can quietly become de facto semantics if contracts are weak
- routing work can spiral into an unbounded graph-layout problem if the milestone is not kept narrow
- density improvements can turn into style churn if not tied to operator tasks
- parity pressure with mature ECAD products can bloat the milestone far beyond an honest next proof

## 9. Recommended Carry-Forward Split

If M12 succeeds, the recommended next ownership split is:

- **M12** - renderer correctness and operator-density hardening
- **M13** - deeper electrical workflow breadth, richer notation behavior, larger proof corpus, or compatibility-oriented expansion

That keeps the roadmap honest:

- first make the current electrical surface trustworthy
- then widen deeper workflow and compatibility scope
