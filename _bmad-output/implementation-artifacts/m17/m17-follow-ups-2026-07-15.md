# M17 Follow-ups

Date: 2026-07-15

## Open follow-up: cross-platform Tree-sitter wasm toolchain bootstrap

### Current state

- `ide/tree-sitter-athena/tree-sitter-athena.wasm` is checked in, so normal Athena product builds do **not** need Zig just to consume Tree-sitter in `ide/`.
- `yarn --cwd ide/tree-sitter-athena build` is the canonical wasm regeneration path.
- That build path now auto-bootstraps a pinned Zig toolchain only on **Windows x64** into repo-local `.tools/zig/`.

### Remaining gap

- The auto-bootstrap path is **not yet implemented** for:
  - macOS x64
  - macOS arm64
  - Linux x64
  - other Linux/macOS runner variants

### Required follow-up

Extend `ide/tree-sitter-athena/scripts/build-tree-sitter-wasm.mjs` so wasm regeneration can bootstrap a pinned, repo-local Zig toolchain on the CI platforms Athena expects to use, without relying on machine-specific hard-coded paths or preconfigured global toolchains.

### Acceptance target

At minimum, the wasm regeneration path should become self-bootstrapping on the main supported CI/desktop platforms:

1. Windows x64
2. macOS arm64
3. macOS x64
4. Linux x64

### Until that follow-up lands

- Athena builds that **consume** the checked-in wasm may run without Zig.
- CI jobs that want to **regenerate** `tree-sitter-athena.wasm` on macOS/Linux must still provide Zig via:
  - `ATHENA_ZIG`
  - `ATHENA_ZIG_BIN`
  - `PATH`

### Ownership rule

This follow-up must not change the M17 parser boundary:

- `ANTLR4` remains the only compiler/LSP parser authority.
- Tree-sitter remains syntax UX only.
- This is a toolchain portability follow-up, not a parser-architecture change.

## Open follow-up: projection/presentation pluginability beyond electrical proof families

### Current state

- The compiler no longer hardcodes `cabinet` as its emitted SVG fallback view.
- View selection now follows approved plugin-published SVG-capable views in deterministic order.
- However, projection notation derivation in `kernel/compiler/src/main/kotlin/com/engineeringood/athena/compiler/ProjectionModelDeriver.kt`
  still contains electrical-family-specific symbol keys such as `device.cabinet.default` and
  `device.wiring.default`.

### Remaining gap

- Future non-electrical domains can publish views and render contributions through the current
  plugin seams.
- They cannot yet publish a fully plugin-owned notation/symbol mapping strategy without core
  changes in the projection/presentation derivation path.

### Required follow-up

Introduce a governed plugin-owned notation/presentation extension seam so symbol keys, label
policies, and family-specific projection notation packs do not require electrical-family logic in
compiler core.

### Acceptance target

At minimum, a future approved domain plugin should be able to:

1. publish one supported view definition
2. publish one render contribution
3. publish its notation/symbol mapping without editing compiler core

### Ownership rule

This follow-up must preserve the same architectural boundaries:

- `Engineering IR` remains canonical semantic authority.
- The compiler may orchestrate extension points, but domain-specific notation policies must move
  out of core-owned electrical switches.
- Tree-sitter remains IDE syntax UX only and is not involved in projection semantic authority.

## Open follow-up: compiler orchestrator decomposition after parser migration

### Current state

- The old handwritten parser monolith is removed.
- `AthenaLanguageParser` is now a small facade over the ANTLR-backed parse engine.
- The next large-class risk is `kernel/compiler/src/main/kotlin/com/engineeringood/athena/compiler/AthenaCompiler.kt`,
  which still owns pass orchestration across parse, lower, validate, backend preparation, backend
  emission, repository helpers, and plugin inventory exposure.

### Remaining gap

- The parser migration removed the biggest syntax-side monolith.
- The compiler orchestration class is still large enough that future feature growth can easily turn
  it into the next maintenance bottleneck.

### Required follow-up

Split `AthenaCompiler` by responsibility so pass orchestration, repository services, and
downstream backend preparation/emission do not continue accumulating in one file.
