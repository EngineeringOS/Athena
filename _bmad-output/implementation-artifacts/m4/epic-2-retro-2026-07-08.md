# Epic 2 Retrospective - M4 - 2026-07-08

## Epic

- Epic 2: Edit Athena Source With Serious Language Tooling

## Outcome

- Epic 2 is complete.
- Athena now routes authored-source diagnostics, completion, document symbols, definition, references, and repeated-edit continuity through the JVM Athena LSP path instead of frontend-owned semantic shortcuts.

## What Went Well

- The semantic authority boundary stayed disciplined: frontend and backend remained transport and workbench layers while `ide/lsp` owned compiler-backed document state.
- The authoring surface stayed intentionally small for M4 while still reaching a credible baseline: diagnostics, completion, symbols, definition, and references.
- Repeated-edit continuity was proven with a direct stdio regression that covered invalid-to-valid recovery and stale lower-version replay.
- Windows verification stayed stable once builds remained sequential and Java 25 was enforced consistently.

## What Hurt

- Epic 2 still needed a late review fix after Story 2.4: the backend transport layer was waiting for an exact diagnostics version even after the LSP server was intentionally changed to reject stale rollback versions and republish the latest tracked version.
- The backend package still has no dedicated automated test harness, so one transport-level invariant had to be proven through build-plus-script verification instead of a package-local test.
- Story closure was again batched near the end of the epic instead of being cleaned up continuously.

## Key Lessons

1. Once Athena LSP becomes the semantic source of truth, every downstream wait condition must follow tracked-document monotonicity rather than frontend request assumptions.
2. Direct stdio proofs are necessary but not sufficient; IDE transport behavior needs its own explicit regression checks when version-sensitive logic changes.
3. Epic 2 succeeded because the feature set stayed narrow. The next workbench story should preserve that discipline and avoid smuggling new semantic authority into UI code.

## Follow-Through For Epic 3

- Build Epic 3 on the existing `frontend -> LSP -> runtime/compiler` boundary without introducing frontend-owned semantic caches.
- Add at least one backend-facing regression proof whenever transport logic changes around document lifecycle or diagnostics versioning.
- Keep the professional workbench effort focused on composition, navigation, and inspection surfaces, not on replacing Athena LSP ownership.

## Action Items

1. Add an explicit backend transport regression proof for stale or superseded diagnostics versions in a future IDE verification pass.
2. Keep story status cleanup continuous during Epic 3 instead of deferring all review-to-done transitions to the epic closeout.
