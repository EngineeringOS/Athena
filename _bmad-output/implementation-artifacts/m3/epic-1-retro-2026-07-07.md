# Epic 1 Retrospective - M3 - 2026-07-07

## Epic

- Epic 1: Prove The Hosted Extensibility Platform

## Outcome

- Epic 1 is complete.
- The workspace now has a dedicated plugin SPI boundary, governed plugin discovery and approval, inspectable hosted lifecycle, and an explicit compiler pass pipeline.

## What Went Well

- The plugin contracts were moved into a real kernel subgroup instead of being left as compiler internals.
- The source-versus-approval split made the architecture materially clearer and easier to test.
- The pass-pipeline refactor stayed aligned with the manifesto rule that orchestration is kernel-owned and domain-neutral.
- Java 25 verification stayed green while the module graph and package roots changed.

## What Hurt

- Story bookkeeping lagged behind code reality; several completed stories remained in `review` or `in-progress`.
- Refactoring plugin infrastructure while also renaming and moving modules created more documentation churn than expected.
- Windows Gradle execution discipline had to be re-learned repeatedly until it was made explicit.

## Key Lessons

1. Stable SPI work must be reflected in the physical directory structure, not only in Gradle aliases.
2. Discovery, approval, and hosted lifecycle need separate tests because they fail in different ways.
3. Sprint-status and story-file status should be closed immediately after successful verification.

## Follow-Through For Next Epic

- Preserve the plugin API and plugin host separation.
- Keep all domain meaning outside kernel modules.
- Treat examples and milestone artifacts as proof assets, not disposable notes.

## Action Items

1. Keep future hosted-boundary changes behind explicit regression tests first.
2. Continue enforcing sequential Java 25 verification on Windows.
