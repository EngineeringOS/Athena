# Epic 2 Retrospective

Date: 2026-07-07
Epic: 2
Status: done

## Epic Review

Epic 2 proved that the desktop-facing multi-view workflow can stay runtime-owned while consuming the explicit M2 projection chain and one command-backed mutation path.

Completed stories:

- 2.1 Host Runtime Projection Sessions And Supported View Switching
- 2.2 Expose Runtime-Owned Projection Snapshots To The Desktop Surface
- 2.3 Refresh Projection State After One Supported Semantic Mutation Path
- 2.4 Preserve History And Diff Meaning Across Projection Refresh
- 2.5 Complete The Desktop Multi-View Operator Proof

Key outcomes:

- Runtime-owned projection sessions now host active view identity and supported view switching.
- The desktop surface consumes runtime-owned snapshots instead of deriving projection truth privately.
- One command-backed mutation path refreshes validation, layout, geometry, rendering, history, and diff consequences through the same runtime path.
- The final desktop operator proof demonstrates semantic continuity across `cabinet` and `wiring` and creates `connection:PLC1.out->M1.in` through the command runtime.

## What Went Well

- The runtime-above-compiler posture held even as view switching, refresh, and inspection became more interactive.
- Selection continuity across multiple views stayed anchored to canonical semantic identity instead of UI-local object state.
- The operator-proof smoke path turned the milestone closeout into a repeatable verification step instead of a manual demo only.

## What Hurt

- Story files and sprint status stayed in `review` longer than they should have after implementation was already green.
- Desktop proof seeding introduced a second kind of M2 example data, which briefly leaked into geometry proof inventory assumptions.
- Some older naming choices in the projection model required a cleanup pass after the epic was functionally complete.

## Lessons Learned

- Command-only mutation remains the right rule for preserving one trustworthy semantic path across DSL, runtime, GUI, and future AI proposals.
- Projection-session ownership belongs in runtime; desktop should present and verify it, not reinvent it.
- Milestone closeout should include status hygiene, naming hygiene, and proof-fixture hygiene before the next epic starts.

## Next Epic Preparation

Preparation completed during and after Epic 2:

- The desktop bootstrap, operator-proof verifier, and smoke tasks now provide a stable Java 25 verification path for the multi-view proof.
- The projection model naming has been simplified across kernel, runtime, compiler, renderer, tests, and docs.
- The M2 sprint tracker now reflects the actual completed state of both epics.

Follow-up actions:

- Carry runtime-owned projection sessions and command-only mutation rules forward unchanged into the next milestone.
- Close reviews and tracker updates continuously instead of deferring them to the end-of-milestone cleanup.
