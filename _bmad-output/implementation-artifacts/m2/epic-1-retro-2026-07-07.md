# Epic 1 Retrospective

Date: 2026-07-07
Epic: 1
Status: done

## Epic Review

Epic 1 established the explicit downstream projection spine for the renumbered M2 milestone: `Engineering IR -> Layout IR -> Geometry IR -> backend/viewer`.

Completed stories:

- 1.1 Establish Explicit Layout And Geometry Model Modules
- 1.2 Contribute The First Supported View Definitions From The Electrical Extension
- 1.3 Derive Layout IR For Cabinet And Wiring Views
- 1.4 Derive Geometry IR And Preserve Canonical Identity Across Projection Layers
- 1.5 Feed Geometry IR To The First Backend And Publish The M2 Proof Corpus

Key outcomes:

- Dedicated `kernel/layout-model` and `kernel/geometry-model` modules now hold the first durable projection contracts.
- The electrical extension contributes the first supported `cabinet` and `wiring` views without taking ownership away from the kernel.
- Layout and geometry derivation are explicit compiler consequences instead of renderer-private or viewer-private structures.
- The first geometry-backed backend proof corpus exists and proves deterministic SVG output from one semantic source through two supported views.

## What Went Well

- The projection split landed as a real kernel boundary, not as UI convenience types.
- Canonical semantic identity remained stable across engineering, layout, and geometry layers.
- The proof corpus kept the milestone concrete and prevented the projection model from becoming abstract architecture only.

## What Hurt

- `IR` appeared in too many model noun names and started to work against readability instead of helping it.
- The proof corpus and the later desktop operator-proof seed now live in the same `examples/m2/` folder, which requires explicit inventory discipline in tests.
- Review closeout lagged implementation, which made the tracker temporarily weaker than the actual repo state.

## Lessons Learned

- Package and module boundaries already communicate the layer; noun types should stay simple unless a collision truly exists.
- Projection proof data and desktop demonstration data can share a milestone folder, but the automation boundaries must stay explicit.
- Once layout and geometry become real kernel contracts, later UI and runtime work gets easier because projection truth is no longer implicit.

## Next Epic Preparation

Preparation completed during and after Epic 1:

- The projection model nouns were simplified so the API surface reads by domain first and layer second.
- The compiler, runtime, renderer, tests, READMEs, and implementation docs now align with the explicit layout and geometry split.
- The geometry proof inventory was tightened to expectation-backed examples only, leaving desktop-only seeds out of backend proof checks.

Follow-up actions:

- Keep model nouns concise and reserve `IR` in names for transformation and validation roles.
- Keep geometry proof-corpus fixtures and desktop proof seeds scope-separated in future test design.
