## Epic Review

### Scope Reviewed

- `1.1` Publish canonical projection contracts in `kernel/projection-model`
- `1.2` Materialize deterministic projection state from engineering layout and geometry inputs
- `1.3` Host runtime-owned projection sessions with deterministic invalidation and refresh
- `1.4` Expose typed projection queries and governed commands through `ide/lsp`

### Outcome

Epic 1 successfully froze the projection boundary without leaking renderer, frontend, or framework authority into kernel semantics.

The completed path is now:

- `kernel/projection-model` owns renderer-neutral projection contracts above layout and geometry
- compiler and runtime derive deterministic projection state from canonical upstream inputs
- runtime owns active projection sessions and invalidation behavior
- `ide/lsp` exposes typed projection queries plus an explicit governed command surface

### Review Verdict

No blocking implementation findings were identified in Epic 1 after review.

Verification evidence:

- `cmd /c "call java25 && .\gradlew.bat --no-daemon --console=plain :kernel:projection-model:test :kernel:compiler:test :kernel:runtime:test :ide:lsp:test"`

### What Worked Well

- The projection boundary stayed above layout and geometry instead of collapsing back into raw engineering entities.
- Runtime-owned session refresh kept stale projection state from surviving in the frontend.
- `ide/lsp` remained a narrow Athena-owned transport seam instead of becoming a graph-framework tunnel.
- Stable semantic identity survived from authored DSL through projection payloads.

### Challenges And Lessons

- Projection contracts are only trustworthy when they derive from the same compiler/runtime path as the rest of the semantic system.
- Session invalidation rules matter early; once frontend-local projection state is allowed to linger, later graphical work becomes fragile.
- Transport seams must stay typed and inspectable, otherwise the projection layer becomes much harder to reason about during debugging.

### Technical Debt And Residual Risk

- Projection queries are proven for the current read-only workbench slice, not yet for future richer editing or review workflows.
- There is still no end-to-end automated desktop interaction test that exercises projection refresh across live repository edits.

## Next Epic Preparation

### Carry-Forward Actions

1. Keep future graphical workflows downstream of `ProjectionSession` and the governed `ide/lsp` protocol.
2. Do not let renderer adapters or Theia widgets acquire projection persistence authority.

Epic 1 successfully proved the M7 architectural base:

- projection boundary frozen

- typed projection protocol proven
