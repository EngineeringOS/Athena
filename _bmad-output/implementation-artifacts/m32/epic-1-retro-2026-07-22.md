# M32 Epic 1 Retrospective: Package Descriptor Contracts

## Scope

Epic 1 implementation reached `review` for:

- Story 1.1: Engineering Package Descriptor v0
- Story 1.2: Representation Package Descriptor v0
- Story 1.3: Representation Descriptor validation

This is an implementation retrospective, not a code-review closeout. Sprint story statuses remain
`review` until the review workflow promotes them.

## Epic Review

Epic 1 established the platform contract layer for M32:

- Engineering Package Descriptor owns reusable engineering/catalog knowledge.
- Representation Package Descriptor owns view/resource package contracts.
- Representation Descriptor owns validated resource usage facts.
- Graphic Resource remains a renderer resource handle, not semantic truth.

## What Worked

- The new `kernel:package-model` module is the right boundary for platform package contracts. It
  avoids burying M32 contracts inside compiler-only knowledge-package infrastructure or renderer
  code.
- TDD caught the intended missing-contract failures before implementation in each story.
- Separate model and validation files kept tiny DTO clusters readable while avoiding mixed
  responsibility files.
- Story evidence improved trust: each story records RED, GREEN, full regression, encoding audit,
  and AC-to-evidence mapping.

## What Needs Attention

- Diagnostics are currently separate per contract family. That is acceptable for v0, but resolver
  stories should decide whether a generic package diagnostic envelope is needed for transport and
  structured product smoke.
- Representation Descriptor validation is contract-level only. Story 4.1 must be careful not to
  bypass it when descriptors begin feeding M30 Representation Occurrences.
- Epic 1 introduced several new contract files quickly. Epic 2 should avoid adding resolver code
  directly into those model files.

## Lessons Learned

1. Package boundaries must be named by authority, not by file/resource format.
2. Descriptor validation must fail before projection success; no renderer fallback can count as
   success.
3. AC-to-evidence in the story file is useful enough to keep mandatory for every M32 story.
4. Documentation needs to explain what each package layer does not own, not only what it owns.

## Next Epic Preparation

Epic 2 should build resolver/runtime mechanics without weakening Epic 1 contracts:

- Registry roots must be explicit and deterministic.
- Resolver facts should identify selected package id, version, descriptor path, validation status,
  dependency facts, selected root, and diagnostics.
- Cache identity must later include descriptor/resource/policy/profile content, but Story 2.1
  should not jump ahead into cache invalidation.
