# M32 Epic 2 Retrospective: Local Registry, Resolver, And Cache

## Scope

Epic 2 implementation reached `review` for:

- Story 2.1: Governed local package registry roots
- Story 2.2: Package resolver and resolution facts
- Story 2.3: Package cache identity and invalidation

This is an implementation retrospective. Story statuses remain `review` until code review promotes
them.

## Epic Review

Epic 2 established the first M32 runtime boundary:

- `:kernel:package-runtime` owns governed registry behavior, resolver facts, and cache identity.
- `:kernel:package-model` remains contract-only.
- Registry discovery uses explicit local roots and does not scan arbitrary workspace folders.
- Resolver facts fail closed for missing, ambiguous, incompatible, or invalid packages.
- Cache identity changes when descriptor, resource, policy, profile, package, or selected-root
  inputs change.

## What Worked

- Adding `:kernel:package-runtime` kept resolver behavior out of descriptor model files.
- Requirement-order diagnostics in the resolver made failures easier to map back to the request.
- Cache identity used standard JDK hashing and avoided new dependencies.
- Tests stayed structured and did not rely on screenshots or renderer behavior.

## What Needs Attention

- Resolver candidates are typed in memory. A later story must introduce descriptor file loading
  carefully, without making file names semantic authority.
- Diagnostics remain family-specific. Epic 3 transport/binding evidence may need a shared envelope.
- Cache identity currently computes keys only; no persistent store or invalidation runtime exists.
  That is intentional, but downstream stories must not claim cache reuse is implemented.

## Lessons Learned

1. Runtime behavior needs its own module boundary as soon as contracts become executable.
2. Deterministic does not always mean alphabetically sorted; traceability can require request-order
   diagnostics.
3. Cache identity should be computed from declared content and policy identities, not raw renderer
   resources.
4. The story-level polish/purge gate is catching boundary drift early.

## Next Epic Preparation

Epic 3 should connect engineering packages, presentation profiles, and representation packages
through explicit profile/manifest/binding contracts:

- Presentation Profile must remain independent policy.
- Binding Manifest must not embed geometry or compiler behavior.
- Binding Resolver should consume Epic 2 facts rather than invent new package lookup paths.
