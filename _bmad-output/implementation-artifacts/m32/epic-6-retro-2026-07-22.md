# M32 Epic 6 Retrospective: Compatibility Cleanup And Milestone Closeout

## Scope

Epic 6 implementation reached `review` for:

- Story 6.1: Consolidate Authoring Preview Compatibility
- Story 6.2: Align Non-Theia Relationship Mutation Surfaces
- Story 6.3: Replace Broad Port Candidate Affordance
- Story 6.4: Resolve Projection Compatibility Fallbacks And Close M32

This is an implementation retrospective. Story statuses remain `review` until code review promotes
them.

## Epic Review

Epic 6 closed the remaining compatibility debt around M31/M32 boundaries:

- `AthenaAuthoringSessionRuntimeService` is retained as explicit
  `legacy-preview-readonly-v1` read-only compatibility with no mutable source authority.
- `AthenaConnectPortsCommand` is retained as explicit
  `legacy-connect-ports-runtime-command-v1` runtime compatibility; product authoring remains
  `SemanticRelationshipIntent`.
- Graphical relationship candidate highlighting no longer treats all `port:` ids as connectable;
  it uses semantic inspection port facts, signal compatibility, direction compatibility, and
  rejected-target evidence.
- The M26 display-title sheet-role fallback was removed from graph-glsp and rebuilt product output.
  Sheet role now comes only from typed `sheet.role` or normalized `policyEvidence.sheetViewRole`.
- `_reference` occurrence strings are documented as legacy defensive fixtures only. Normal
  compiler, runtime, and LSP projection payloads prove duplicate visual reference components are
  not emitted.

## What Worked

- Versioning retained compatibility APIs made cleanup auditable instead of relying on informal
  legacy comments.
- Test-first graph-glsp coverage caught the exact display-title fallback behavior before removal.
- Fixed-string scans were useful after CodeGraph to distinguish product source from generated or
  legacy fixture hits.
- The M32 sample product smoke and density proof survived the cleanup and remained the baseline for
  customer-demo credibility.

## What Needs Attention

- Generated desktop product bundles can hold stale adapter code after source changes; future
  stories that touch Theia-visible adapters should rebuild or explicitly purge those bundles during
  the final gate.
- `_reference` defensive fixtures should eventually be renamed to a clearer legacy-fixture naming
  scheme so they cannot be mistaken for production projection output.
- Each future story should keep the final polish/purge task as mandatory, including stale generated
  output, duplicate sprint keys, open action items, and AC-to-evidence checks.

## Lessons Learned

1. Compatibility should either be removed or named with a versioned contract; unnamed fallbacks
   become hidden architecture.
2. Visible text is not projection authority. Display names are presentation, not sheet-role policy.
3. Legacy fixtures are acceptable only when normal product payloads have stronger no-regression
   tests proving the production path.
4. Product-demo integrity depends on both source tests and generated runtime artifacts being fresh.
