# M32 Epic 5 Retrospective: Customer Demo And Product Proof

## Scope

Epic 5 implementation reached `review` for:

- Story 5.1: Create M32 Sample Packages And Project
- Story 5.2: Add Structured Package Product Smoke
- Story 5.3: Polish Professional Demo Layout And Density

This is an implementation retrospective. Story statuses remain `review` until code review promotes
them.

## Epic Review

Epic 5 turned the package platform contracts into a customer-demo proof:

- `examples/m32/sample-project` now contains Athena-owned semantic source, package descriptors,
  profiles, manifests, representation descriptors, vector resources, and layout proof data.
- `M32SamplePackageSet` gives package-runtime tests a deterministic sample boundary without adding
  parser, renderer, or `.athena` visual syntax authority.
- `M32ProductSmokeProofRunner` proves package resolution, binding, descriptor validation,
  occurrence creation, rendering payloads, route anchoring, profile switching, and no fallback.
- `M32DemoLayoutDensityProofRunner` proves transparent normal chrome, no duplicate labels,
  descriptor-driven anchors/labels, compact composition, tight viewBox, and sheet-navigation-safe
  layout.

## What Worked

- Structured proof kept customer-demo claims auditable without relying on screenshots as authority.
- The sample stayed synthetic and Athena-owned, avoiding QET/vendor/runtime licensing ambiguity.
- Profile switching was proven as Binding Resolver behavior, not `.athena` source mutation.
- Density proof caught the exact visual problems that triggered M30/M32: visible wrappers,
  duplicated labels, generic fallback boxes, and hard-coded oversized viewBox.

## What Needs Attention

- The IDE/Theia display path still needs to consume these structured proof payloads in later product
  polish; Epic 5 deliberately did not let Theia become package authority.
- Screenshot evidence remains secondary and should be captured only after the structured proof is
  stable.
- Epic 6 cleanup must not remove compatibility paths in a way that silently breaks the M32 sample
  proof.

## Lessons Learned

1. Customer demo credibility should be tested as structured product evidence first and screenshots
   second.
2. Sample package data belongs under examples and package-runtime fixtures; it is not semantic
   source syntax.
3. A small number of synthetic product-like packages can prove the platform boundary better than a
   large weak symbol library.
4. ViewBox and chrome rules need explicit assertions; otherwise wrapper UI drifts back in.
