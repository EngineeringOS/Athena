# M32 Epic 3 Retrospective: Presentation Profile And Binding Resolver

## Scope

Epic 3 implementation reached `review` for:

- Story 3.1: Presentation Profile v0
- Story 3.2: Binding Manifest v0
- Story 3.3: Binding Resolver selection
- Story 3.4: Binding Evidence product payloads

This is an implementation retrospective. Story statuses remain `review` until code review promotes
them.

## Epic Review

Epic 3 established the policy and binding layer between Engineering Packages and Representation
Packages:

- `PresentationProfileDescriptor` owns independent appearance policy.
- `BindingManifest` remains compatibility data and does not select geometry or mutate source.
- `BindingResolver` owns deterministic selection and mapping.
- `BindingEvidencePayload` exposes product-safe resolver evidence for proof tooling and adapters.
- Failures preserve authority-specific diagnostics instead of accepting renderer fallback success.

## What Worked

- Keeping Presentation Profile, Binding Manifest, Binding Resolver, and Binding Evidence as
  separate contracts prevented one object from becoming an EPLAN-style macro.
- Runtime behavior stayed in `:kernel:package-runtime`; `:kernel:package-model` remained
  descriptor/manifest contract-only.
- TDD made the missing boundaries obvious: unresolved resolver and payload symbols failed before
  implementation.
- Product payloads resolved the Epic 2 question about binding evidence by introducing a
  binding-specific transport-safe envelope rather than a broad cross-family diagnostic type.

## What Needs Attention

- Binding Resolver currently consumes in-memory descriptors and package descriptors. Epic 4 must
  connect this to Representation Occurrence creation without turning descriptor ids or resource ids
  into semantic truth.
- The resolver has v0 version-range behavior. It is enough for M32 proof, but richer semver policy
  should stay behind package-runtime when needed.
- LSP/Theia integration is documented as adapter-only, but actual frontend consumption is deferred
  to descriptor-backed presentation stories.

## Lessons Learned

1. Binding Manifest should stay boring. Resolver behavior belongs in runtime, not in manifest
   contracts.
2. Presentation Profile switching is the proof that source remains semantic-only.
3. Evidence payloads should be plain transport facts; frontend adapters should not inspect
   renderer resources for package meaning.
4. Authority diagnostics are more useful than generic projection failure once package layers exist.

## Next Epic Preparation

Epic 4 should integrate package-resolved descriptors into M30 representation, renderer, and routing
flows:

- Consume Binding Resolver evidence before creating Representation Occurrences.
- Preserve semantic subject identity through descriptor-backed occurrences.
- Keep renderer and Theia adapter-only.
- Reject center fallback and generic boxes when descriptors, anchors, labels, or resources are
  missing.
