# M31 Epic 1 Retrospective: Trustworthy Engineering Authoring Transactions

## Epic Review

### Outcome

Epic 1 completed all three stories. Athena now discovers authoring capability through the existing
M29 registry, creates exactly one mutable intent per revision-bound transaction, validates that
transaction through an explicit ordered lifecycle, and preserves distinct stale, blocked,
cancelled, committed, reprojected, and projection-failed outcomes.

### What Worked

- Extending `SemanticCapabilityRegistry` prevented a second action-authority model.
- Test-first API design made the transaction envelope and lifecycle boundaries concrete before
  runtime implementation.
- A known UTF-8 SHA-256 assertion caught the difference between proving exact content and merely
  proving digest length.
- Injected validation, revision, mutation-handoff, and reprojection boundaries proved ordering and
  no-call guarantees without pulling source serialization or Theia into Epic 1.
- Sequential Gradle verification avoided the Windows cache/result corruption seen in prior work.
- Per-story CodeGraph review and the M31 cleanup ledger made retained compatibility paths explicit.

### Challenges And Root Causes

- The existing preview session, source mutation evaluator, and new transaction runtime have nearby
  responsibilities. Their boundaries were not obvious from names alone; CodeGraph caller analysis
  was required to distinguish preview state, source evaluation, and accepted mutation orchestration.
- Runtime tests initially lacked direct test visibility of `interaction-model` types because
  `authoring-model` uses an implementation dependency. A test-scoped internal module dependency was
  required; no external dependency was introduced.
- Weak evidence can look complete. Story 1.2 initially asserted only SHA-256 length, which did not
  prove exact UTF-8 hashing. The final evidence now uses a known digest and changed-content check.

### Key Lessons

1. Capability discovery must precede transaction creation; otherwise transaction runtime becomes a
   second policy engine.
2. Revision safety requires semantic snapshot, source URI, document version, and exact content
   digest together. Any subset leaves a stale-acceptance gap.
3. Validation order is part of the platform contract, not an implementation detail. Tests must
   assert stage order and authority call counts.
4. Commit and projection are separate outcomes. Projection failure after commit must retain mutation
   evidence and must not pretend canonical mutation failed or fallback rendering succeeded.
5. Compatibility paths are acceptable only when owner, reason, target story, and verification are
   recorded at the moment they are retained.
