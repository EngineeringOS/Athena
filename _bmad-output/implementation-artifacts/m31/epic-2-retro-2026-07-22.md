# M31 Epic 2 Retrospective: Governed Semantic Model Mutation Authority

## Epic Review

### Outcome

Epic 2 completed all four stories. Athena now has generic semantic entity contracts, semantic-only
Engineering Concept Templates, backend-owned source edit planning, governed nested-port entity
creation, and governed semantic relationship creation. The legacy `ConnectPortsIntent` product path
was retired in favor of `SemanticRelationshipIntent`, and accepted source mutation now passes through
Revision Guard, backend planning, Mutation Authority, compile, and reprojection evidence instead of
frontend/source shortcuts.

### What Worked

- The Epic 1 transaction envelope held up. Stories 2.3 and 2.4 reused the same lifecycle instead of
  creating entity-only or relationship-only transaction machinery.
- Concept templates stayed semantic. The electrical motor template lives in the electrical extension
  and carries nested ports without importing representation primitives, SVG, coordinates, or viewBox.
- Backend source planning became a real authority seam. Entity creation, relationship creation, and
  authored layout now use parsed AST/source-span planning and parser-admitted text.
- BMAD review caught authority gaps that normal happy-path tests missed: client edit rejection,
  stale update acceptance, lifecycle collapse, relationship diagnostic loss, and runtime graph
  side effects before durable source planning.
- The mandatory polish/purge gate removed real stale paths, including legacy relationship transport
  names, frontend-local relationship authority, and obsolete generated frontend artifacts.

### Challenges And Root Causes

- Several flows initially returned an accepted-looking payload before the durable mutation had
  actually happened. The root issue was treating "source edit returned to the client" as equivalent
  to "Mutation Authority committed". Story 2.3 corrected that boundary.
- Some tests asserted rejected status but not unchanged state. The final graph-command review proved
  that a rejected authored-layout source plan could still move runtime projection placement until a
  no-side-effect assertion was added.
- Relationship capability evidence initially focused on the source subject. Story 2.4 had to bind
  evidence to both canonical endpoints, active source context, origin, and domain requirements.
- Stale names are easy to leave behind when generated TypeScript output is present. Source scans must
  distinguish active source authority from ignored build artifacts, then purge stale generated files
  where possible.

### Key Lessons

1. A governed authoring flow is not complete until failed and rejected paths prove zero mutation and
   zero downstream side effects.
2. Returning an editor edit is not a commit. Commit only exists after Revision Guard validation,
   authoritative source mutation, compile, and lifecycle update.
3. Source planning must happen before any runtime placement or projection side effect whenever the
   user action claims durable authored source.
4. Capability evidence must describe the whole semantic action, not only the initiating subject.
   Relationship authoring needs both endpoints and domain compatibility evidence.
5. Generated build output is not architecture authority, but stale generated artifacts still confuse
   cleanup reviews and should be purged during the story polish gate.
