# M31 Epic 4 Retrospective: Complete Graphical Authoring Workflow

## Epic Review

### Outcome

Epic 4 completed all three stories. Athena's Graphical View can initiate governed entity creation,
render preview and lifecycle evidence, accept backend-owned source edits, create governed semantic
relationships, reveal canonical subjects, preserve sheet/mode controls, expose distinct lifecycle
diagnostics, and keep nested Outline ports visible after reopen.

### What Worked

- The frontend stayed an adapter. Source edit planning, relationship compatibility, lifecycle state,
  and reveal identity come from backend/platform contracts.
- Review patches found real async and stale-state bugs: hidden create previews, stale relationship
  responses, source-edit offset mismatch, invalid selector caching, and lifecycle collapse.
- Focused frontend tests gave fast feedback while preserving the full product smoke for Epic 5.
- The UI remained dense by avoiding persistent wrapper chrome and by treating screenshots as
  secondary evidence.

### Challenges And Root Causes

- Graphical workflows have more race surfaces than kernel contracts. Preview, editor revision,
  selected source, selected target, sheet mode, and backend responses can all change independently.
- Some older affordances still exist for compatibility, especially broad `port:` candidate
  highlighting before backend preview. The backend authority is correct, but candidate discovery UX
  is not fully capability-evidence-first yet.
- Non-Theia CLI/desktop/Compose surfaces still use the older low-level command path and require a
  later multi-surface alignment milestone.

### Key Lessons

1. Frontend acceptance must recheck current editor, current preview token, source URI, document
   version, and edit offsets immediately before applying returned edits.
2. Lifecycle states must be rendered as distinct states; `projection-failed` after commit is not the
   same as a rejected or blocked mutation.
3. Reveal should target canonical subject identity returned by the platform, not labels, DOM nodes,
   endpoint guesswork, or route geometry.
4. Compatibility affordances can remain only when they are ledgered and not represented as M31
   acceptance behavior.

### Quality Evidence

- Stories 4.1, 4.2, and 4.3 are marked done with review findings resolved.
