# Epic 2 Retrospective

Date: 2026-07-06
Epic: 2
Status: done

## Epic Review

Epic 2 proved that semantic inspection, mutation, history, diff, recomputation, and runtime-hosted extension points can all stay inside one runtime-owned path without giving up canonical ownership.

Completed stories:

- 2.1 Expose The Engineering Graph Over The Active Project
- 2.2 Introduce The Command Runtime For Semantic Mutations
- 2.3 Record Command History With Undo, Redo, Replay, And Serialization
- 2.4 Deliver One GUI Command-Backed Port Connection Mutation Path
- 2.5 Recompute Affected Scope And Refresh Validation And Rendering
- 2.6 Expose Semantic Diff And History Inspection
- 2.7 Host Runtime Plugins For Domain Semantics, Commands, And Views
- 2.8 Enforce Non-Sovereign Plugin Boundaries
- 2.9 Route Optional AI Proposals Through Accepted Commands

Key outcomes:

- The engineering graph exists as a runtime projection over canonical semantic state.
- Semantic mutation now flows through explicit commands instead of caller-side direct edits.
- Undo, redo, replay, diff, and history inspection are available over runtime-managed changes.
- Runtime-hosted plugins extend domain semantics, commands, and views without becoming sovereign owners.
- Optional AI proposals are constrained to accepted command-shaped changes instead of direct mutation.

## What Went Well

- The runtime-centered model held up as more interactive concerns were added.
- Command, history, diff, and recomputation stayed explainable against canonical identities.
- Non-sovereign plugin rules remained enforceable while still allowing real extension behavior.

## What Hurt

- BMAD tracking lagged implementation and left many completed stories parked in `review`.
- Repository grouping and naming clarity needed a second pass after the first logical-only refactor.
- Active documentation and historical documentation were briefly mixed in ways that made the current topology harder to read.

## Lessons Learned

- Command-only mutation is the right discipline for keeping GUI, DSL, and AI paths aligned.
- Physical repository structure matters; architectural grouping should be visible in the actual tree, not only in Gradle identifiers.
- Review closeout needs to happen continuously or sprint status stops reflecting the real project state.

## Next Epic Preparation

Preparation completed during and after Epic 2:

- Module layout is now grouped under `kernel`, `extensions`, `ui`, and `apps`.
- Root and module READMEs exist for the active grouped topology.
- Historical compiler notes are now labeled so they do not masquerade as current taxonomy.

Follow-up actions:

- Carry non-sovereign plugin rules and command-only mutation rules unchanged into the next epic.
- Keep project tracking synchronized with real implementation milestones instead of batching status cleanup at the end.
